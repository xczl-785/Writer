## 你的任务

### 阶段一：现状探索与诊断

> 你没有历史上下文。请像一个首次接手项目的架构师一样，自主探索仓库全貌。

1. **自主探索仓库的完整目录结构**（`tree`、`ls`、查看关键文件），不限定你的阅读范围
2. **理解项目的文档体系**：哪些是冻结的规格文档？哪些是活跃的工作文档？它们之间的关系是什么？
3. **理解项目的代码现状**：是否已有代码骨架？技术栈配置是否就位？
4. **输出诊断报告**，包含：
   - 当前目录树（`tree` 形式）
   - 问题清单（按严重度排序）
   - 每个问题附带一句话说明"为什么这对 AI Agent 协作是障碍"
   - 诊断结论：当前结构能否支撑 AI Agent Teams 直接进入开发？

### 阶段二：目标结构设计（基于阶段一的诊断结论）

> 仅当阶段一诊断出需要调整时才执行此阶段。如果当前结构基本合理，只需给出微调建议。

设计目标结构时，遵循以下原则：

#### 核心原则

| 原则 | 说明 |
|------|------|
| **AI 可读性优先** | 目录命名、文件命名必须自描述，减少歧义 |
| **单一入口** | 必须有一个明确的入口文件，AI 读完它就能建立全局地图 |
| **冻结 vs 活跃分离** | 已决策的规格文档 与 正在演进的工作文档 必须物理隔离 |
| **产出物有归属** | 每类产出（文档、代码、测试、脚本、配置）都有明确的存放位置 |
| **约定优于配置** | 使用业界通用的目录约定（如 `src/`、`tests/`、`docs/`、`scripts/`） |
| **扁平优于嵌套** | 在清晰度允许的前提下，减少目录嵌套层级 |

#### 必须覆盖的层次

1. **文档层** — 规格文档、架构文档、决策记录、任务管理
2. **代码层** — 源码组织，符合项目技术栈的惯例
3. **测试层** — 测试文件的组织方式
4. **脚本层** — 构建、部署、开发辅助脚本
5. **配置层** — 项目配置、CI/CD、环境配置

#### 输出

- 目标目录结构树（`tree` 形式，带注释说明每个目录/文件的用途）
- 每个调整点标注：`[必须在开发前完成]` 或 `[可与开发并行]`
