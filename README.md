# Writer 项目前言（Phase 0）

## 项目定位
本项目处于原始需求分析阶段。当前目标不是立刻开发，而是把原始想法细化为：
- 可执行的功能列表
- 明确的功能边界
- 可验证的验收标准

产品方向：
- 保留 Typora 风格核心体验（所见即所得、极简、流畅输入）
- 提供 Obsidian 风格的基础目录能力（应用内新建文件、目录管理、快速切换）
- 启动速度与轻量化优先

## 当前阶段角色分工
- 你：产品 Owner（目标、优先级、验收拍板）
- AI 模型：实现与测试执行者（不越权改范围）
- 助手（我）：需求分析与技术约束顾问（把模糊需求转成可执行条目）

## 当前任务（Now）
1. 形成 V1 功能清单（必须做 / 可延后 / 明确不做）
2. 定义 V1 边界（只做写作核心，不做复杂生态）
3. 定义“像 Typora 一样好用”的验收标准（可观察、可测试）
4. 输出第一版 PRD（简版）

## 关键问题：功能列表和边界从哪里来？
不是只“对比竞品抄功能”，而是四个来源合并，按权重决策。

### 来源 1：你的真实写作流程（最高权重）
从你的高频动作提取功能：
- 打开应用 -> 新建/切换文档 -> 连续写作 -> 导出（可选）
原则：高频流程卡顿点，优先级最高。

### 来源 2：体验对标（Typora/Obsidian）
- Typora：定义“编辑体验基线”
- Obsidian：仅借用“目录/文件管理能力”
原则：按“能力拆解”对标，不按“功能总量”对标。

### 来源 3：技术与成本约束
- 跨平台（macOS/Windows）
- 启动速度与包体
- 单人 + AI 协作开发成本
原则：任何会显著拖慢启动速度或增加复杂度的功能，默认降级优先级。

### 来源 4：验收可测性
每个候选功能都要回答：
- 是否可在 1-2 句话描述验收标准？
- 是否能在一次回归里被稳定验证？
不能稳定验收的功能，不进入 V1。

## 边界判定规则（用于是否纳入 V1）
某功能同时满足以下 3 条，则进入 V1：
1. 直接服务高频写作流程
2. 不明显破坏极简与流畅
3. 可以被明确验收

若只满足 1 条或 2 条，进入 Backlog；
若引入高复杂度生态（如插件平台、图谱、协作系统），默认不进 V1。

## 决策口径（一句话）
不是“能不能做”，而是“是否提升无打断写作主流程”。
