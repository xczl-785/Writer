# 菜单遗漏说明

> 更新日期：2026-03-13  
> 用途：记录当前菜单系统中的遗漏项，并区分“当前菜单修复范围”与“需单独立项的需求开发范围”。

## 1. 使用说明

本文档只解决三个问题：

1. 当前有哪些菜单项已经暴露，但能力未闭环。
2. 哪些项应当作为当前菜单修复任务直接处理。
3. 哪些项虽然也属于遗漏，但开发量较大，应单独作为需求开发，不纳入当前修复批次。

## 2. 当前修复原则

### 2.1 直接修复

满足以下条件的功能，纳入当前菜单修复：

- 已经在菜单中展示。
- 目标能力边界清晰。
- 不需要单独设计完整的新交互模式。
- 不涉及新的文档模型或新的导出体系。

### 2.2 单点置灰优先

以下菜单项当前不支持，优先按“单点置灰”处理，而不是继续保留可点击占位态：

- `File > Save As`
- `File > Export PDF`
- `File > Export HTML`
- `File > Export Image`
- `View > Source Mode`

说明：

- 这些项暂不进入当前功能补齐范围。
- 目标是先让界面表达准确，避免“能点但无功能”。

### 2.3 单独需求开发

满足以下任一条件的功能，不按当前菜单修复处理，应单独立项：

- 需要新增编辑模型或扩展体系。
- 需要新增完整交互方案。
- 需要新增导出、视图模式、渲染或序列化体系。
- 开发量明显高于普通菜单接线。

## 5. 需要单独需求开发的内容

以下内容不建议作为当前菜单修复的一部分直接实现。

### 5.1 Math Block

所属菜单：

- `Paragraph > Math Block`

建议结论：

- 单独需求开发

原因：

- 不只是补一个菜单命令。
- 需要确定数学公式块的数据结构、编辑态、渲染态与 Markdown 表达方式。
- 需要补齐公式输入、渲染、序列化、选区行为、删除行为等完整能力。

开发量说明：

- 明显高于 `任务列表`、`Link`、`Underline` 这类菜单补齐项。
- 至少涉及以下决策：
  - 数学扩展选型
  - LaTeX 输入与渲染方案
  - Markdown 解析与序列化规则
  - 编辑器节点行为
  - 样式与交互规范

结论：

- 该项应先补需求文档，再进入设计与实现。

### 5.2 Save As / Export 系列

所属菜单：

- `File > Save As`
- `File > Export PDF`
- `File > Export HTML`
- `File > Export Image`

建议结论：

- 单独需求开发

原因：

- 涉及文件路径选择、导出格式、异常处理和结果定义。
- 并非单纯的编辑器内部命令。
- `Export PDF` 尤其可能涉及额外渲染与版式策略。

### 5.3 Source Mode

所属菜单：

- `View > Source Mode`

建议结论：

- 单独需求开发

原因：

- 实际是视图模式能力，不是单点菜单修补。
- 涉及源码视图与富文本视图之间的切换语义、同步策略和交互边界。

## 6. 关键说明

### 6.1 任务列表为什么优先

- 用户已明确感知该问题。
- Slash 菜单和顶部菜单都受影响。
- 根因明确，修复路径清晰。
- 修复后可以同时校正菜单层、编辑器层、Markdown 层的一致性。

### 6.2 数学公式为什么不能顺手补

- 它不是“菜单项没接线”，而是“完整能力不存在”。
- 若直接在当前批次里硬做，极容易把需求、设计、实现混在一起，后续返工概率高。

## 7. 当前建议顺序

### 7.1 当前修复顺序

1. 置灰 File 菜单中的暂不支持项
2. 置灰 View 菜单中的 `Source Mode`
3. 任务列表
4. Link
5. Image
6. Highlight
7. Underline
8. Slash 图片链路对齐

### 7.2 单独立项顺序

1. Math Block
2. Source Mode 完整能力
3. Save As
4. Export PDF / HTML / Image
