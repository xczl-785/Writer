## 3. 本轮已确认决策

### 3.1 空工作区（S4）与无工作区（S5）必须严格区分

本轮已明确确认：

> **空工作区（S4）与无工作区 / 空白态（S5）必须作为两个不同的一等状态处理，不得继续混用。**

对应定义如下：

| 状态 | 定义 | 是否属于工作区上下文 |
|---|---|---|
| S4 空工作区 | 当前仍处于某个已保存工作区中，但该工作区当前不包含任何根文件夹 | 是 |
| S5 无工作区 / 空白态 | 当前不存在工作区上下文 | 否 |

该确认意味着后续方案、实现与测试必须显式覆盖这两个状态，而不能再仅以“folders 是否为空”进行粗粒度判断。

### 3.2 S4 的产品表现要求已确认

当系统处于 **S4 空工作区** 时，已确认以下口径：

1. 当前工作区仍然存在，不能退化为“无工作区”；
2. 状态栏应继续显示当前工作区名称；
3. File 菜单中以下动作应保持可用：
   - `保存工作区`
   - `工作区另存为`
   - `将文件夹添加到工作区`
   - `关闭工作区`
4. `关闭文件夹`应为不可用状态；
5. 主区应显示“当前工作区为空”的专属空态表达，而不是普通欢迎页/空白页。

### 3.3 S5 的产品表现要求已确认

当系统处于 **S5 无工作区 / 空白态** 时，已确认以下口径：

1. 当前不存在工作区上下文；
2. 状态栏不显示工作区名称；
3. File 菜单中的工作区相关动作应按“无工作区”规则禁用；
4. 主区显示普通欢迎页 / 空白页表达；
5. S5 与 S4 在命名、菜单可用性、主区表达上均不得混淆。

### 3.4 File 菜单中的“关闭文件夹”目标对象已确认

本轮已明确确认：

> **当当前工作区包含多个根文件夹时，`File > 关闭文件夹` 默认关闭当前选中的根文件夹。**

该确认同时约束了该动作的对象语义：

1. `关闭文件夹`是成员级动作，而非容器级动作；
2. 该动作的目标对象应来自“当前选中的根文件夹”，而不是任意根文件夹；
3. 该动作不得被实现为“直接关闭整个工作区”；
4. 若当前不存在可被关闭的目标根文件夹，则该动作应禁用，而不是退化为其他动作。

### 3.5 后续推进方式已确认：先溜边，再完整实施

本轮已明确确认以下推进原则：

> **后续在制作计划时，应优先采用“溜边形式”，先将不确定内容、当前逻辑不支持内容、状态断层与语义缺口规划落地，再进入完整方案与正式开发。**

该推进原则的目标包括：

1. 避免零碎内容在后续被遗漏；
2. 避免当前实现中的历史逻辑持续干扰新需求判断；
3. 先显式暴露边界问题，再进行整体设计与编码；
4. 为后续测试边界、迁移风险和回归点梳理提供基础材料。

### 3.6 S5 下“将文件夹添加到工作区”已确认禁用

本轮已明确确认：

> **在 S5 无工作区 / 空白态下，`将文件夹添加到工作区` 应禁用。**

确认理由如下：

1. S4 与 S5 的核心差异在于是否已存在工作区容器；
2. `将文件夹添加到工作区` 属于容器内成员操作；
3. 若 S5 中仍允许该动作，会弱化“当前并不存在工作区”的边界；
4. 因此应保持：
   - **S4：可用**
   - **S5：禁用**

### 3.7 `关闭文件` 已纳入本轮 File 菜单重塑范围

本轮已明确确认：

> **`关闭文件` 属于本轮 File 菜单重塑范围，不作为后续版本再单独收口。**

该确认意味着后续本轮菜单设计中，需一并处理 `关闭文件` 的：

1. 菜单位置；
2. 可用性规则；
3. 与工作区状态并存时的语义边界；
4. 与 `关闭文件夹`、`关闭工作区` 之间的层级区分。

### 3.8 `关闭文件夹` 的执行前提已进一步收紧

本轮已明确确认：

> **`File > 关闭文件夹` 仅在“当前明确选中根文件夹”时才允许执行；若当前选中的是子目录、文件或其他对象，则该动作应禁用。**

这意味着：

1. `关闭文件夹` 的对象解析不得再做“所属根文件夹”的隐式上推；
2. 该动作只接受“当前选中的根节点”作为合法目标；
3. 该规则有助于稳定成员级动作的对象边界，避免文件级/目录级/根级对象混淆。

### 3.9 隐式工作区是当前阶段的总体设计方向

本轮已明确确认：

> **Writer 当前阶段应采用“隐式工作区”方向：底层统一为工作区模型，表层表达仍以文件夹、文件操作为主。**

对应口径如下：

1. 底层模型上，用户始终处于某个工作区上下文或无工作区状态；
2. 表层表达上，不强行向用户灌输“工作区优先”的显式心智；
3. 用户主感知对象仍应优先是：
   - 文件夹
   - 文件
   - 当前文件操作
4. 仅在涉及上下文持久化、容器识别与容器级动作时，才显式呈现“工作区”概念。

### 3.10 S1 下保留“保存工作区”与“工作区另存为”

本轮已明确确认：

> **在 S1 单文件夹临时工作区下，`保存工作区` 与 `工作区另存为` 均保留且可用。**

确认理由如下：

1. 该能力符合“打开文件夹 → 沉淀为可复用上下文”的自然演进路径；
2. 该能力支撑“隐式工作区”的底层统一，不要求用户先显式理解工作区；
3. 对用户而言，入口仍是“打开文件夹”，工作区保存能力是对当前上下文的进阶持久化，而非强制主流程。

### 3.11 S1 下“关闭文件夹”与“关闭工作区”均保留且可用

本轮已明确确认：

> **在 S1 单文件夹临时工作区下，`关闭文件夹` 与 `关闭工作区` 均保留且可用。**

同时确认：

1. 二者在执行结果上都可回到 S5；
2. 但在产品语义上仍必须保留区分：
   - `关闭文件夹`：成员级动作
   - `关闭工作区`：容器级动作
3. 不得因为当前阶段结果相同，就将两者定义为完全同义动作。

### 3.12 S3 单根已保存工作区下关闭唯一文件夹后必须进入 S4

本轮已明确确认：

> **当处于 S3 已保存工作区，且当前仅包含一个根文件夹时，执行 `关闭文件夹` 后必须进入 S4 空工作区，不得直接回到 S5。**

确认理由如下：

1. 一旦工作区已保存，容器身份即已成立；
2. `关闭文件夹` 的对象是成员，不是容器；
3. 移除唯一成员后应保留容器，并进入“空工作区”状态；
4. 只有执行 `关闭工作区` 时，才允许退出到 S5。

### 3.13 S5 下 File 菜单工作区相关动作规则已确认

本轮已明确确认：

在 **S5 无工作区 / 空白态** 下：

#### 可用动作

- `打开文件`
- `打开文件夹`
- `打开工作区`
- `打开最近`

#### 禁用动作

- `保存工作区`
- `工作区另存为`
- `将文件夹添加到工作区`
- `关闭文件夹`
- `关闭工作区`
- `关闭文件`

其中，`关闭文件` 在 S5 下禁用的理由为：

1. S5 下不存在当前活动文件；
2. 该动作属于文件级关闭动作，不应在无文件上下文时保持可执行。

### 3.14 `关闭文件` 的作用边界已确认

本轮已明确确认：

> **`关闭文件` 指向当前活动文件；无活动文件时禁用；关闭最后一个文件后仍保留当前工作区上下文。**

对应口径如下：

1. `关闭文件` 是文件级动作；
2. 其目标对象始终为当前活动文件；
3. 若没有活动文件，则动作禁用；
4. 关闭最后一个文件后，不额外创造新的复杂状态，只保留当前工作区上下文。

### 3.15 Open Recent 的分组规则已确认

本轮已明确确认：

> **`打开最近` 采用稳定三组结构：工作区 / 文件夹 / 文件，并坚持唯一归属规则。**

对应口径如下：

1. 分组顺序固定：
   - 工作区
   - 文件夹
   - 文件
2. 分组语义固定；
3. 同一对象只能归属一个分组；
4. 空组允许隐藏，但不改变整体分组顺序语义。

### 3.16 S4 空工作区主区动作层级已确认

本轮已明确确认：

在 **S4 空工作区** 下，主区空态动作层级应为：

1. **主动作**：`将文件夹添加到工作区`
2. **次动作**：`关闭工作区`

同时确认：

1. `打开文件夹`
2. `打开工作区`

这两个动作可继续保留在菜单路径中，但不作为 S4 主区空态主按钮呈现。

### 3.17 `打开文件夹` 与 `将文件夹添加到工作区` 的关系已确认

本轮已明确确认：

> **`打开文件夹` 与 `将文件夹添加到工作区` 均保留，但二者语义必须明确区分。**

对应定义如下：

| 动作 | 语义 |
|---|---|
| `打开文件夹` | 进入或替换当前上下文 |
| `将文件夹添加到工作区` | 修改当前工作区成员 |

该确认是“底层统一，表层自然”设计原则的重要组成部分。

## 4. 对当前代码理解下的直接约束

基于当前代码现状，本轮确认项对后续实现具有以下直接约束意义：

### 4.1 工作区状态模型不能再仅使用 `folders.length`

后续状态建模必须能表达以下差异：

- `有已保存工作区，但当前 0 个根文件夹`；
- `当前根本不存在工作区上下文`。

因此，后续实现中不得继续仅将 `folders.length === 0` 视为统一的空态。

### 4.2 状态栏命名逻辑必须显式覆盖 S4

后续状态栏命名逻辑必须支持：

- 单文件夹临时工作区：显示文件夹名；
- 单文件夹已保存工作区：显示工作区名；
- 多文件夹未保存工作区：显示“未命名工作区（未保存）”；
- 已保存多文件夹工作区：显示工作区名；
- 空工作区：显示工作区名；
- 无工作区：不显示工作区标识。

### 4.3 File 菜单动作可用性必须按状态细分

后续 File 菜单不能再仅用“是否有文件夹”判断工作区动作可用性。至少需要区分：

- S1 单文件夹临时工作区；
- S2 多文件夹未保存工作区；
- S3 已保存工作区；
- S4 空工作区；
- S5 无工作区 / 空白态。

### 4.4 “关闭文件夹”需要依赖当前根文件夹选择态

若后续实现 `File > 关闭文件夹`，则其执行前提应为：

1. 当前存在选中的根文件夹；
2. 该选中对象必须是根文件夹，而不是普通子目录或文件；
3. 不允许将子目录、文件等对象隐式上推映射为所属根文件夹；
4. 若当前选择不满足根节点条件，则该动作应处于不可用状态或无法执行。

### 4.5 后续实现应遵循“底层统一，表层自然”

后续方案与实现应遵循以下统一原则：

1. 内部状态与对象关系尽量统一为工作区模型；
2. 表层用户表达仍优先使用文件夹、文件等自然对象；
3. 不为追求内部纯度而将“工作区”概念过度前置给用户；
4. 仅在持久化、容器识别与容器级动作中显式呈现工作区语义。

### 4.6 File 菜单需显式覆盖文件级 / 成员级 / 容器级三层动作

基于本轮确认，后续 File 菜单至少需稳定表达以下三层动作：

1. **文件级动作**
   - 打开文件
   - 关闭文件
2. **成员级动作**
   - 打开文件夹
   - 关闭文件夹
   - 将文件夹添加到工作区
3. **容器级动作**
   - 打开工作区
   - 保存工作区
   - 工作区另存为
   - 关闭工作区

该分层是后续菜单命名稳定性与禁用态推导的基础。

## 7. 当前确认摘要

截至 2026-03-15，本轮已确认的核心执行口径为：

1. **S4 空工作区** 与 **S5 无工作区** 必须严格分开；
2. **S4 仍然是合法工作区容器状态**，需保留工作区名称、动作可用性与专属空态表达；
3. **S5 不是工作区状态**，应使用普通欢迎页 / 空白态表达；
4. **`File > 关闭文件夹` 在多根场景下默认关闭当前选中的根文件夹**；
5. **`File > 关闭文件夹` 仅在当前明确选中根文件夹时可执行**；
6. **当前阶段采用“隐式工作区”方向：底层统一，表层自然**；
7. **S1 下保留 `保存工作区`、`工作区另存为`、`关闭文件夹`、`关闭工作区`**；
8. **S3 单根已保存工作区关闭唯一文件夹后必须进入 S4**；
9. **S5 下 `将文件夹添加到工作区`、`关闭文件夹`、`关闭工作区`、`关闭文件` 均禁用**；
10. **`关闭文件` 纳入本轮，且其对象固定为当前活动文件**；
11. **Open Recent 采用“工作区 / 文件夹 / 文件”稳定三组与唯一归属规则**；
12. **S4 主区主动作固定为“将文件夹添加到工作区”，次动作固定为“关闭工作区”**；
13. **`打开文件夹` 与 `将文件夹添加到工作区` 均保留，但分别承担“替换上下文”与“修改成员”语义**；
14. **后续计划应先采用“溜边式规划”补齐边界与缺口，再进入完整方案与开发。**
