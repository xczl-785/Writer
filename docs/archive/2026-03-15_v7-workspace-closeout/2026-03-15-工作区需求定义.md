## 2. 文档目标

本文档用于将“工作区概念定义”转化为可执行的产品需求约束，明确当前阶段必须满足的对象关系、状态规则、菜单规则、状态栏规则与关闭行为规则。

本文档回答的问题是：

1. 产品必须支持哪些工作区状态；
2. 各状态下允许哪些核心动作；
3. 菜单、状态栏与关闭行为需要遵循什么规则；
4. 哪些内容属于当前阶段范围，哪些不属于。

---

## 3. 范围说明

### 3.1 本文档范围内

1. 单文件夹、多文件夹、已保存工作区、空工作区的产品需求；
2. 文件夹与工作区的对象关系；
3. File 菜单相关动作的需求约束；
4. 状态栏命名与状态识别需求；
5. “关闭文件夹”“关闭工作区”的行为需求；
6. 空工作区的合法性与可恢复性需求。

### 3.2 本文档范围外

1. 具体 UI 样式、视觉布局与控件尺寸；
2. 动效、提示方式、弹窗文案细节；
3. 工程实现方案、代码结构与存储实现；
4. 自动化测试方案与技术拆分。

## 4. 依赖概念

本需求文档基于以下概念前提：

1. 工作区是统一抽象；
2. 单文件夹属于临时工作区；
3. 多文件夹属于工作区的显性形态；
4. 已保存工作区具有独立稳定身份；
5. 空工作区是已保存工作区的一种合法状态。

若后续概念定义文档发生调整，应同步评估本文档。

---

## 5. 产品对象与状态需求

### 5.1 对象层级需求

产品需明确区分以下对象层级：

| 层级 | 对象 | 说明 |
|---|---|---|
| 成员层 | 根文件夹 | 工作区中的成员对象 |
| 容器层 | 工作区 | 承载当前工作上下文的容器对象 |

需求约束：

1. 文件夹不得被定义为与工作区互斥的对象；
2. 文件夹应被视为工作区中的成员；
3. 工作区应被视为包含 0~N 个根文件夹的容器。

### 5.2 状态支持需求

产品当前阶段必须支持以下状态：

| 状态 | 必须支持 |
|---|---|
| 单文件夹临时工作区 | 是 |
| 多文件夹未保存工作区 | 是 |
| 已保存工作区 | 是 |
| 空工作区 | 是 |

### 5.3 状态迁移需求

产品必须支持以下状态迁移：

1. 打开文件夹 → 单文件夹临时工作区；
2. 单文件夹临时工作区 + 添加文件夹 → 多文件夹未保存工作区；
3. 单文件夹临时工作区 / 多文件夹未保存工作区 + 保存工作区 → 已保存工作区；
4. 已保存工作区移除全部根文件夹 → 空工作区；
5. 空工作区添加文件夹 → 已保存工作区。

## 6. 状态栏需求

### 6.1 命名规则需求

状态栏必须承担“当前工作上下文标识”的职责，命名规则如下：

| 场景 | 状态栏显示要求 |
|---|---|
| 单文件夹临时工作区 | 显示根文件夹名称 |
| 单文件夹且已保存为工作区 | 显示工作区名称 |
| 多文件夹未保存工作区 | 显示“未命名工作区（未保存）” |
| 多文件夹已保存工作区 | 显示工作区名称 |
| 空工作区 | 显示工作区名称 |

### 6.2 约束要求

1. 状态栏显示不得简单等同于内部数据结构字段；
2. 状态栏命名应优先服务于用户对当前上下文的识别；
3. 空工作区不得退化为“无上下文”显示；
4. 已保存工作区场景下，优先显示工作区名称，而非根文件夹名称。

## 7. File 菜单需求

### 7.1 菜单结构需求

File 菜单当前阶段需围绕以下对象组织：

1. 文件
2. 文件夹
3. 工作区
4. 导出
5. 关闭类动作

### 7.2 基础动作需求

当前阶段 File 菜单需支持以下能力表达：

- 打开文件
- 打开文件夹
- 打开工作区
- 打开最近
- 将文件夹添加到工作区
- 保存工作区
- 工作区另存为
- 导出
- 关闭文件
- 关闭文件夹
- 关闭工作区

### 7.3 状态化展示需求

File 菜单中的相关动作需根据当前上下文状态进行合理组织。

需求约束如下：

1. File 菜单命名应保持稳定，不因状态变化而频繁改名；
2. File 菜单结构应保持稳定，不建议通过显隐改变核心动作布局；
3. 当前状态差异应优先通过“可用 / 不可用”表达，而非通过菜单项消失表达；
4. 单文件夹临时工作区场景下，“关闭文件夹”与“关闭工作区”均应可用；
5. 单文件夹已保存工作区场景下，“关闭文件夹”与“关闭工作区”均应可用；
6. 多文件夹场景下，“关闭文件夹”与“关闭工作区”均应可用；
7. 空工作区场景下，“关闭文件夹”应不可用，“关闭工作区”应可用；
8. 无工作区 / 空白态场景下，“关闭文件夹”“关闭工作区”“保存工作区”“工作区另存为”应不可用；
9. 空工作区场景下，“将文件夹添加到工作区”应保持可用。

### 7.4 菜单稳定性需求

为保持对象语义和用户记忆一致性，当前阶段 File 菜单需遵循以下原则：

1. 菜单动作名称尽量固定；
2. 菜单项顺序尽量固定；
3. 用户应能够通过禁用态理解“该动作存在，但当前上下文不满足执行条件”；
4. 当前阶段不建议将“关闭文件夹”改名为“移除文件夹”等新命名；
5. 当前阶段不建议将“保存工作区”“关闭工作区”等动作按状态改写为不同标题。

## 8. 关闭行为需求

### 8.1 关闭文件夹需求

“关闭文件夹”必须始终被定义为：

> 面向根文件夹成员的操作

其需求要求如下：

1. 在多文件夹场景下，关闭文件夹应仅移除目标根文件夹，不关闭工作区；
2. 在单文件夹临时工作区场景下，关闭文件夹可导致返回空白态；
3. 在单文件夹已保存工作区场景下，关闭文件夹应导致进入空工作区，而非关闭工作区；
4. 在空工作区场景下，不应存在“关闭文件夹”的可执行对象。

### 8.2 关闭工作区需求

“关闭工作区”必须始终被定义为：

> 面向工作区上下文容器的操作

其需求要求如下：

1. 关闭工作区应直接结束当前工作区上下文；
2. 关闭工作区不应被表达为“移除某个文件夹”；
3. 无论当前工作区包含 0、1 还是多个根文件夹，关闭工作区的目标对象始终是容器本身。

### 8.3 单文件夹临时工作区需求

在单文件夹临时工作区场景下：

1. “关闭文件夹”与“关闭工作区”可在结果上表现一致；
2. 但产品定义中仍需保留两者的语义差异；
3. 后续文案与交互表达不得将二者宣称为完全同义动作。

### 8.4 单文件夹已保存工作区需求

在单文件夹已保存工作区场景下：

1. “关闭文件夹”与“关闭工作区”必须在结果上分离；
2. “关闭文件夹”后的目标状态必须为空工作区；
3. “关闭工作区”后的目标状态必须为无工作区 / 空白态；
4. 不允许将“关闭文件夹”隐式转换为“关闭工作区”。

### 8.5 多文件夹场景需求

在多文件夹场景下：

1. “关闭文件夹”与“关闭工作区”必须严格区分；
2. 关闭文件夹不得影响其他根文件夹的继续存在；
3. 关闭工作区必须整体结束当前上下文；
4. 当多文件夹工作区经过连续移除后仅剩一个根文件夹时，后续行为应切换到单文件夹规则，而不是继续沿用多文件夹规则。

## 9. 空工作区需求

### 9.1 合法性需求

空工作区必须被定义为合法状态，而非异常状态。

### 9.2 用户价值需求

空工作区需体现以下产品价值：

1. 保持已保存工作区的稳定身份；
2. 支持用户先保留工作区容器，再重新组织成员；
3. 支持工作区作为“可编辑容器”而非“仅在非空时存在的集合”。

### 9.3 动作可用性需求

空工作区场景下至少应支持以下动作：

- 将文件夹添加到工作区
- 打开文件夹
- 保存工作区
- 工作区另存为
- 关闭工作区

### 9.4 空态表达需求

空工作区必须具有与“无工作区 / 空白态”明确区分的页面表达。

需求约束如下：

1. 空工作区页面必须明确告知用户：当前仍处于某个工作区中；
2. 空工作区页面不得以错误态、异常态或失败态文案进行表达；
3. 空工作区页面应将“恢复当前工作区内容”作为首要引导目标；
4. 空工作区页面不应将“离开当前工作区”作为唯一主路径。

### 9.5 空态动作层级需求

空工作区场景下，动作层级需求如下：

1. **主动作**必须为：`将文件夹添加到工作区`；
2. **次动作**应支持：`关闭工作区`；
3. `打开文件夹`可以保留在菜单路径中，但不要求作为空态主区域主动作；
4. 空工作区场景下不得继续提供“关闭文件夹”动作。

### 9.6 状态提示需求

空工作区场景下的状态提示需满足以下要求：

1. 状态栏继续显示工作区名称；
2. 页面主区域承担对“空工作区”状态的主要说明职责；
3. 状态栏不要求重复承载过强说明信息；
4. 产品表达应体现：工作区为空，不等于当前无工作区。

## 10. 非目标

当前阶段明确不作为本轮产出的内容包括：

1. 工作区视觉原型的最终样式稿；
2. 空工作区的详细空态界面方案；
3. 代码实现与持久化细节；
4. 与工作区无直接关系的其他菜单整理项。

## 11. 最近打开需求

### 11.1 分组需求

`文件 > 打开最近` 当前阶段应采用稳定分组结构，包含以下三组：

1. **工作区**
2. **文件夹**
3. **文件**

### 11.2 唯一归属需求

最近打开中的各类对象应采用**唯一归属**规则，不再采用重复归属策略。

归属要求如下：

| 对象类型 | 最近打开归属 |
|---|---|
| 已保存工作区 | 工作区组 |
| 空工作区对应的已保存工作区 | 工作区组 |
| 单文件夹临时工作区 | 文件夹组 |
| 普通最近打开文件夹 | 文件夹组 |
| 最近打开文件 | 文件组 |

### 11.3 约束要求

1. 单文件夹临时工作区不得同时出现在“工作区组”和“文件夹组”；
2. 当前阶段不再沿用“双重归属”策略；
3. 同一个对象在同一个最近打开菜单中不得重复展示；
4. 工作区组应优先表达“具名、可持久化的工作区对象”；
5. 文件夹组应优先表达“文件夹入口历史”；
6. 文件组应优先表达“文件入口历史”。

### 11.4 数量需求

当前阶段建议沿用以下容量约束：

| 分组 | 数量上限 |
|---|---:|
| 工作区组 | 10 |
| 文件夹组 | 5 |
| 文件组 | 20 |

### 11.5 菜单表达需求

最近打开菜单应满足以下表达要求：

1. 三个分组结构保持稳定；
2. 已保存工作区以工作区名称作为主识别信息；
3. 文件夹组以文件夹名称作为主识别信息；
4. 文件组以文件名作为主识别信息；
5. 当存在重名项时，应补充必要的路径信息以辅助区分。

---

## 12. 需求结论

当前阶段关于工作区的产品需求可概括为：

1. 产品必须承认工作区是统一容器抽象；
2. 产品必须承认单文件夹是工作区的一种状态；
3. 产品必须承认空工作区是已保存工作区的一种合法状态；
4. 产品必须始终将“关闭文件夹”与“关闭工作区”定义为不同层级的动作；
5. 产品必须通过菜单、状态栏与场景规则体现上述分层，而不得在语义上混淆成员与容器；
6. 最近打开必须采用“工作区 / 文件夹 / 文件”的稳定分组结构；
7. 最近打开中的对象必须采用唯一归属，不再重复出现在多个分组中。
