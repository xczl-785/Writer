## 1. 讨论结论

### 1.1 范围定义

**决策**: C. 综合推进型

> 同时推进交互设计与结构感知两条主线，但如果结构感知对交互设计依赖度高且交互设计耗时，可将结构感知推迟至 V6。

**条件判断**:

- 交互设计优先
- 结构感知视依赖情况决定纳入 V5 或推迟 V6

---

### 1.2 时间策略

**决策**: C. 弹性周期

> 不设硬性时间要求，以功能完成度为标准。
> **核心原则**: 把交互设计的基础打扎实，为将来扩展打好基础。
> **约束**: 交互太重要，不能草率。

---

### 1.3 "基础"定义

**决策**: C. 两者结合（基础设施 + 核心功能）

**工作顺序**:

```
顶层设计 → 基础设施 → 核心功能
    │           │          │
    ▼           ▼          ▼
软件风格    框架/规范    具体交互
交互风格    系统        实现
用户故事
```

### 3.3 验收框架（已确认）

**决策**: D. 混合型 — 三维度验收框架

```
┌─────────────────────────────────────────────────────────┐
│  用户故事 (价值导向)                                      │
│  - 定义"做什么"                                          │
│  - 最高评分权重                                          │
│  - 格式：As a... I want to... So that... + 验收条件      │
├─────────────────────────────────────────────────────────┤
│  功能清单 (落地保障)                                      │
│  - 定义"怎么做"                                          │
│  - 驱动交互设计实际落地                                   │
│  - 格式：功能项 + 技术验收点                              │
├─────────────────────────────────────────────────────────┤
│  质量属性 (验收底线)                                      │
│  - 定义"做多好"                                          │
│  - 作为设计上的验收准则                                   │
│  - 格式：质量指标 + 度量标准                              │
└─────────────────────────────────────────────────────────┘
```

**原则**: 单独任意一个方面都是不合格的，必须三维度结合。

## 4. 关键发现

### 4.1 设计理念参考（来自 Gemini 提案）

**核心理念："Invisible Fluidity" (隐形流动)**

- **Premium (高级感)**：精密仪器质感，动效克制
- **Focus (专注)**：内容是主角，UI 弱化融入背景
- **Tactile (触感)**：操作有物理反馈感

### 4.2 竞品交互模式

| 产品         | 核心交互特点                                       |
| ------------ | -------------------------------------------------- |
| **Typora**   | 混合模式（输入时渲染，光标移入展开源码），极致沉浸 |
| **Obsidian** | 左侧装订线，结构清晰，适合极客                     |
| **Notion**   | 块操作（六个点拖拽柄），极强的结构调整能力         |
