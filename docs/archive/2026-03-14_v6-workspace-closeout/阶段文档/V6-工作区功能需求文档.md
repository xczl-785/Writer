## 2. 执行摘要

### 2.1 背景

当前 Writer 采用单文件夹工作区模式，用户只能打开一个文件夹作为工作区。根据竞品调研，Typora 因仅支持单文件夹模式被用户诟病（GitHub issue #2126 高票请求），而 Obsidian 的 Vault 模型过于刚性。V6 版本将引入完整的工作区系统，支持单文件夹和多文件夹工作区。

**竞品参考策略**: 优先参考 iA Writer 和 Zettlr 的简洁性，而非 VS Code 的 IDE 范式，以匹配写作工具用户的心智模型。

### 2.2 目标

1. **核心目标**: 支持多文件夹工作区，允许用户将多个独立目录组合成一个工作区
2. **体验目标**: 符合用户心智模型，降低学习成本，保持轻量体验
3. **产品目标**: 保持"沉静、专注、可信赖"的产品人格，不引入过度复杂的配置系统

### 2.3 范围

**包含**:

- 单文件夹工作区（规范化现有能力）
- 多文件夹工作区（多个独立目录共同构成一个工作区）
- 文件夹拖拽打开/添加工作区
- 最近工作区/文件历史
- 工作区状态持久化

**不包含** (V6 不做):

- 工作区级别的全局搜索（跨文件夹搜索）— 建议 V6.1 考虑
- 工作区模板/预设
- 工作区共享与同步
- 工作区级别的扩展/插件配置
- 远程工作区（网络路径）
- 工作区级别的快捷键定制
- 内部拖拽移动文件（高风险，V6.1 考虑）
- 工作区设置（设置层集成复杂，V6.1 考虑）
- 单文件打开与单文件拖拽（转移至 V6.1 独立需求）

### 2.4 版本节奏建议

根据战略顾问建议，采用分阶段发布策略：


## 3. 产品定位对齐

### 3.1 产品人格

Writer 的产品人格定义为：**沉静、专注、可信赖的写作伙伴**。

| 维度   | 本功能设计含义                 | 校验                                                                |
| ------ | ------------------------------ | ------------------------------------------------------------------- |
| 沉静   | 工作区切换无声执行，不制造噪音 | ✅ 非模态通知优先，状态栏被动呈现                                   |
| 专注   | 让内容成为焦点，工具隐形       | ✅ 空状态简洁，无多余信息；多文件夹为进阶能力，不干扰初级用户       |
| 可信赖 | 工作区状态自动恢复，行为可预测 | ✅ 状态持久化，边缘情况有明确处理（文件损坏、权限不足、路径不存在） |

### 3.2 设计原则校验

本功能设计遵循以下原则：

| 原则                     | 校验                                                                              |
| ------------------------ | --------------------------------------------------------------------------------- |
| 无打断 (Non-Disruption)  | ✅ 非模态通知优先；无标题工作区关闭时使用状态栏提示而非模态对话框                 |
| 一致性 (Consistency)     | ✅ 相同操作在不同入口行为一致；快捷键与通用惯例一致（Cmd+O 打开，Cmd+W 关闭）     |
| 可发现 (Discoverability) | ✅ 提供从"可见入口"到"快捷操作"的学习路径；空状态界面提供明确引导                 |
| 渐进式 (Progressive)     | ✅ 单文件夹为默认，多文件夹为进阶能力；术语简化，用户可见层面统一使用"工作区"概念 |

### 3.3 术语简化

为降低用户认知负担，对用户可见的术语进行简化：

| 内部术语       | 用户可见术语 | 说明                                     |
| -------------- | ------------ | ---------------------------------------- |
| 单文件夹工作区 | 工作区       | 普通用户无需理解"单文件夹"概念           |
| 多文件夹工作区 | 工作区       | 仅在窗口标题用"(工作区)"标识             |
| 无标题工作区   | 未保存工作区 | 仅在状态栏提示时使用                     |
| 已保存工作区   | 工作区       | 用户感知为"已保存的工作区"               |
| 根文件夹       | 文件夹       | 文件树中统一显示为文件夹，不强调"根"概念 |

## 4. 用户故事

### 4.1 用户角色

| 角色       | 描述                         | 核心需求                           | V6 功能映射                       |
| ---------- | ---------------------------- | ---------------------------------- | --------------------------------- |
| 深度写作者 | 长文、学术、创作场景         | 快速进入写作状态，不受文件管理干扰 | WS-11 状态恢复，WS-12 空状态界面  |
| 知识管理者 | 多项目并行，需要跨文件夹编辑 | 灵活组织多个相关文件夹             | WS-03 多文件夹，WS-05/06 保存打开 |
| 轻量记录者 | 快速启动、低门槛记录         | 简单直观的文件打开方式             | WS-01 打开文件夹，WS-10 最近文件  |

### 4.2 核心用户故事

#### US-1: 打开文件夹开始写作

**作为** 一名写作者  
**我想要** 打开一个文件夹开始写作  
**以便于** 我能快速访问我的文档

**验收标准**:

- [ ] 用户可以通过"文件 > 打开文件夹"菜单打开文件夹
- [ ] 用户可以通过快捷键 `Cmd+O` (macOS) / `Ctrl+O` (Windows) 打开文件夹
- [ ] 打开后文件树显示该文件夹及其子文件夹下的所有 Markdown 文件（递归显示）
- [ ] 窗口标题显示文件夹名称
- [ ] 如已有工作区打开，提示用户保存当前工作区（如有未保存变更）

**边缘情况**:

- [ ] 文件夹内无 Markdown 文件时，显示空状态提示
- [ ] 文件夹权限不足时，显示错误提示并提供"在访达中显示"选项

---

#### US-2: 组合多个文件夹为工作区

**作为** 一名知识管理者  
**我想要** 将多个文件夹组合成一个工作区  
**以便于** 我能在一个界面中同时编辑项目文档和查阅参考资料，减少上下文切换时间（预计节省 30%）

**验收标准**:

- [ ] 用户可以通过"文件 > 添加文件夹到工作区"添加第二个文件夹
- [ ] 添加后文件树显示多个根文件夹，每个文件夹可独立展开/折叠
- [ ] 窗口标题显示"（工作区）"标识
- [ ] 两个文件夹的文件都可访问和编辑
- [ ] 同名文件夹自动追加路径后缀以示区分（如"docs (项目 A)"和"docs (项目 B)"）

**边缘情况**:

- [ ] 同一文件夹不能重复添加，尝试添加时提示"该文件夹已在工作区中"
- [ ] 达到 10 个文件夹上限时，提示"工作区文件较多可能影响性能"

**作为** 一名深度写作者  
**我想要** 快速访问最近的工作区和文件

- [ ] 支持右键移除单条历史记录
- [ ] 支持"清除所有历史记录"

**数据归属规则**:

- 单文件夹工作区同时出现在"工作区组"和"文件夹组"

**作为** 一名写作者  
**我想要** 我的工作区状态自动保存  
**以便于** 我重新打开 Writer 时能恢复到上次的工作状态

**验收标准**:

- [ ] 关闭 Writer 时自动保存当前工作区状态（打开的文件、侧边栏状态）
- [ ] 重新打开时自动恢复到上次的工作区
- [ ] 上次打开的文件保持打开状态，光标位置恢复
- [ ] 侧边栏展开/折叠状态恢复
- [ ] 可在设置中关闭"启动时恢复上次工作区"

**边缘情况**:

- [ ] 恢复时发现文件已被删除，提示用户并跳过该文件
- [ ] 工作区文件夹已被移动，提示用户重新选择路径

**作为** 一名知识管理者  
**我想要** 管理已保存的工作区文件  
**以便于** 我能清理不再使用的工作区配置，或重新定位已移动的工作区

**验收标准**:

- [ ] 用户可以在 Finder/Explorer 中定位 `.writer-workspace` 文件（右键菜单"在访达中显示"）
- [ ] 删除 `.writer-workspace` 文件不影响实际文件夹
- [ ] 最近工作区列表中可右键移除已删除的工作区记录
- [ ] 工作区文件被移动后，可从最近列表重新打开，系统自动更新路径
- [ ] 支持重命名工作区文件（在 Finder/Explorer 中重命名，Writer 自动适配）

**边缘情况**:

- [ ] 工作区文件被外部删除 → 最近列表中保留记录，打开时提示"工作区文件不存在，是否移除记录？"
- [ ] 工作区文件被移动 → 打开时自动更新路径，提示用户"工作区文件已移动到新位置"

#### US-9: 外部文件变化感知（新增）

**作为** 一名使用 Git 同步或云存储的用户  
**我想要** 系统自动检测外部文件变化  
**以便于** 我能及时处理冲突，避免数据丢失

- 文件夹在外部被添加/删除/移动

**验收标准**:

- [ ] 文件夹被外部删除时，提示"文件夹已被删除"
- [ ] 用户可选择"重新加载"、"比较变更"或"忽略"

### 5.2 功能详细说明

#### 5.2.1 WS-01: 打开单文件夹工作区

**功能描述**: 用户打开一个文件夹作为单文件夹工作区

**用户操作**:

- "文件 > 打开文件夹"菜单
- 快捷键 `Cmd+O` (macOS) / `Ctrl+O` (Windows/Linux)
- 空状态界面"打开文件夹"按钮

**系统行为**:

1. 弹出系统文件夹选择对话框（仅允许选择文件夹）
2. 用户选择后，调用 `list_tree` 加载文件树（递归显示子文件夹）
3. 设置当前工作区为选中路径
4. 清空编辑区状态，准备新工作区
5. 如已有工作区打开，检查未保存变更并提示

**验收标准**:

- [ ] 只能选择文件夹，不能选择文件
- [ ] 文件树仅显示 Markdown 文件（`.md`, `.markdown`, `.mdx`）
- [ ] 跳过常见排除目录（`.git`, `node_modules`, `dist` 等）
- [ ] 加载大量文件时显示进度提示
- [ ] 窗口标题显示文件夹名称
- [ ] 文件夹内无 Markdown 文件时显示空状态提示
- [ ] 文件夹权限不足时显示错误提示

**依赖**: 无

**边缘情况**:

- 文件夹内无 Markdown 文件 → 显示空状态提示
- 文件夹权限不足 → 显示错误提示并提供"在访达中显示"选项
- 已有工作区打开 → 提示保存当前工作区（如有未保存变更）

---

#### 5.2.2 WS-02: 关闭工作区（优先级提升为 P0）

**功能描述**: 关闭当前工作区，返回空状态界面

**用户操作**:

- "文件 > 关闭工作区"菜单
- 快捷键 `Cmd+W` (macOS) / `Ctrl+W` (Windows/Linux) — **与关闭文件一致**
- 空状态界面"关闭"按钮（如有工作区打开）

**系统行为**:

1. 检查未保存文件（dirty 状态）
2. 如有未保存文件，显示保存提示（非模态状态栏通知）
3. 用户确认后，清空所有工作区状态
4. 显示空状态界面

**验收标准**:

- [ ] 关闭前检查未保存文件
- [ ] 未保存文件提示使用状态栏非模态通知（符合无打断原则）
- [ ] 清空工作区状态（folders, openFiles, activeFile）
- [ ] 显示空状态界面
- [ ] 快捷键 `Cmd/Ctrl+W` 与关闭文件一致

**依赖**: WS-01, WS-12

**边缘情况**:

- 无未保存文件 → 直接关闭，无提示
- 有未保存文件但用户选择"不保存" → 记录选择，本次会话不再提示
- 工作区为无标题多文件夹 → 提示"是否保存工作区配置？"（状态栏非模态）

### 5.2.15 WS-19: 外部文件变化感知（新增 P1）

**功能描述**: 自动检测工作区文件/文件夹在外部发生的变化

**技术实现**:

- 使用 `chokidar` 或 `@parcel/watcher` 监听文件系统变化
- 支持 macOS (FSEvents)、Windows (ReadDirectoryChangesW)、Linux (inotify)
- 配置排除模式，避免监听大文件夹（如 `node_modules`、`.git`）

**监听事件**:

- `add` - 新文件/文件夹被添加
- `change` - 文件被修改
- `unlink` - 文件被删除
- `unlinkDir` - 文件夹被删除
- `rename` - 文件/文件夹被移动

**系统行为**:

| 事件             | 文件状态                   | 用户提示                           | 操作选项                        |
| ---------------- | -------------------------- | ---------------------------------- | ------------------------------- |
| 文件被外部修改   | 编辑器未打开               | 静默刷新文件树                     | 无                              |
| 文件被外部修改   | 编辑器已打开且无变更       | 静默重新加载                       | 无                              |
| 文件被外部修改   | 编辑器已打开且有未保存变更 | 状态栏提示"文件已被外部修改"       | [比较变更] [重新加载] [忽略]    |
| 文件被外部删除   | 文件树中                   | 标记"（已删除）"，右键可移除       | [从工作区移除]                  |
| 文件被外部删除   | 编辑器已打开               | 标签页显示"（已删除）"，保存时提示 | [另存为] [忽略]                 |
| 新文件被添加     | 文件夹内                   | 自动出现在文件树中                 | 无                              |
| 文件夹被外部删除 | 工作区根文件夹             | 灰显提示"（已删除）"               | [从工作区移除] [重新定位]       |
| 文件夹被外部移动 | 工作区根文件夹             | 提示"文件夹已被移动"               | [重新定位文件夹] [从工作区移除] |

**验收标准**:

- [ ] 文件被外部修改时，在 1 秒内检测到变化
- [ ] 编辑器已打开且有变更时，显示冲突对话框
- [ ] 对话框提供"比较变更"选项（diff 视图）
- [ ] 文件被删除时，文件树标记为"（已删除）"
- [ ] 新文件被添加时，自动刷新文件树
- [ ] 文件夹被删除/移动时，提示用户重新定位或移除
- [ ] 支持配置排除模式（`files.watcherExclude`）
- [ ] 大文件夹（>1000 文件）显示性能警告

**性能优化**:

- 排除 `.git`、`node_modules`、`dist`、`.venv` 等目录
- 使用防抖避免频繁事件触发（300ms）
- 大工作区使用懒加载 + 虚拟滚动

**依赖**: WS-01

**边缘情况**:

- 文件正在保存时被外部修改 → 保存完成后重新检测
- 网络文件夹同步延迟 → 使用轮询 fallback（5 秒间隔）
- 文件权限变更 → 提示"无权限访问文件"
- 多个文件同时变化 → 批量提示，避免连续弹窗

**设置项**:

```json
{
  "files.watcherExclude": [
    "**/node_modules/**",
    "**/.git/**",
    "**/dist/**",
    "**/.venv/**"
  ],
  "files.autoRefresh": true,
  "files.confirmExternalChanges": true
}
```

## 6. 业务规则

### 6.1 工作区类型

| 类型           | 定义                        | 配置文件                 | 关闭时行为                 | 用户可见术语 |
| -------------- | --------------------------- | ------------------------ | -------------------------- | ------------ |
| 单文件夹工作区 | 只有一个根文件夹            | 无                       | 直接关闭，无提示           | 工作区       |
| 无标题工作区   | 多个文件夹但未保存          | 无（临时状态）           | 状态栏非模态提示"是否保存" | 未保存工作区 |
| 已保存工作区   | 有 `.writer-workspace` 文件 | `.writer-workspace` 文件 | 直接关闭，无提示           | 工作区       |

### 6.2 工作区文件格式

```json
{
  "version": 1,
  "folders": [
    {
      "path": "docs",
      "name": "文档"
    },
    {
      "path": "../notes",
      "name": "笔记"
    }
  ],
  "state": {
    "openFiles": ["docs/readme.md"],
    "activeFile": "docs/readme.md",
    "sidebarVisible": true,
    "sidebarExpandedPaths": ["docs", "docs/api"]
  }
}
```

**版本管理**:

- `version` 字段用于未来格式升级
- 解析时验证 JSON Schema，损坏时优雅降级

### 6.3 路径规则

| 规则            | 说明                                                         |
| --------------- | ------------------------------------------------------------ |
| 相对路径        | 相对于 `.writer-workspace` 文件位置解析                      |
| 支持 `..`       | 支持向上级目录（如`../notes`）                               |
| 绝对路径兼容    | 支持绝对路径，但保存时优先转换为相对路径                     |
| 跨卷/挂载点处理 | 无法转换为相对路径时使用绝对路径，并提示用户可能影响可移植性 |
| 路径分隔符      | 统一使用 `/`，内部自动转换平台特定格式（Windows `\` → `/`）  |

### 6.4 文件数量限制

| 项目             | 限制           | 理由     | 超限处理                           |
| ---------------- | -------------- | -------- | ---------------------------------- |
| 工作区文件夹数量 | 软性上限 10 个 | 性能考虑 | 超过时显示性能警告，但允许继续添加 |
| 最近工作区数量   | 最多 10 条     | UI 简洁  | 超出时自动移除最旧记录             |
| 最近文件数量     | 最多 20 条     | UI 简洁  | 超出时自动移除最旧记录             |
| 单文件夹文件数   | 无硬性限制     | -        | 超过 1000 文件时懒加载 + 性能警告  |

### 6.5 最近项目数据归属规则

| 项目类型                 | 归属                | 说明                                           |
| ------------------------ | ------------------- | ---------------------------------------------- |
| `.writer-workspace` 文件 | 工作区组            | 显式保存的多文件夹工作区                       |
| 单文件夹工作区           | 工作区组 + 文件夹组 | 同时出现在两组，方便不同习惯用户查找           |
| 单个文件                 | 文件组              | 全局最近打开的 Markdown 文件，不限于当前工作区 |

## 11. 边缘情况汇总

### 11.1 文件与路径相关

| 边缘情况                 | 触发条件        | 系统行为                                               | 优先级 |
| ------------------------ | --------------- | ------------------------------------------------------ | ------ |
| 文件夹内无 Markdown 文件 | 打开文件夹      | 显示空状态提示"该文件夹内无 Markdown 文件"             | P0     |
| 文件夹权限不足           | 打开/添加文件夹 | 显示错误提示，提供"在访达中显示"选项                   | P0     |
| 工作区文件损坏           | 打开工作区文件  | JSON Schema 验证失败，提示损坏，提供"创建新工作区"选项 | P1     |
| 文件夹路径不存在         | 打开工作区      | 灰显提示"（不存在）"，右键可移除                       | P1     |
| 跨平台路径解析失败       | 移动工作区文件  | 使用绝对路径并警告可能影响可移植性                     | P1     |
| 工作区文件版本不兼容     | 打开工作区文件  | 提示版本不兼容，提供升级/降级选项                      | P1     |

### 11.2 并发与冲突相关

| 边缘情况             | 触发条件           | 系统行为                                     | 优先级 |
| -------------------- | ------------------ | -------------------------------------------- | ------ |
| 多实例同时打开工作区 | 第二个 Writer 实例 | 提示"该工作区已在其他窗口打开"，提供切换选项 | P1     |
| 工作区文件被外部修改 | 其他程序修改文件   | 提示"工作区文件已被修改，是否重新加载？"     | P1     |
| 打开的文件已被删除   | 恢复状态时发现     | 提示用户并跳过该文件                         | P1     |
| 工作区文件夹已被移动 | 恢复状态时发现     | 提示用户重新选择路径                         | P1     |

### 11.3 性能相关

| 边缘情况               | 触发条件           | 系统行为                                        | 优先级 |
| ---------------------- | ------------------ | ----------------------------------------------- | ------ |
| 文件夹包含大量文件     | 单文件夹>1000 文件 | 懒加载 + 性能警告"文件夹较大，加载可能需要时间" | P1     |
| 工作区文件夹超过 10 个 | 添加第 11 个文件夹 | 显示性能警告，但允许继续添加                    | P1     |
| 状态文件过大           | 打开大量文件       | 仅恢复最近 50 个文件，提示用户部分文件未恢复    | P2     |

### 11.4 用户操作相关

| 边缘情况             | 触发条件           | 系统行为                                         | 优先级 |
| -------------------- | ------------------ | ------------------------------------------------ | ------ |
| 拖拽非 Markdown 文件 | 拖入文件           | 提示"不支持的文件类型"                           | P1     |
| 拖拽文件夹到编辑区   | 拖入文件夹         | 已有工作区时不接受放置，空状态时打开为新工作区     | P2     |
| 重复添加同一文件夹   | 添加已存在文件夹   | 提示"该文件夹已在工作区中"                       | P1     |
| 无标题工作区关闭     | 关闭多文件夹工作区 | 状态栏非模态提示"是否保存工作区配置"             | P1     |

### 12.6 验证计划增强

| 指标                        | 验证时间          | 成功标准 | 失败处理                                  | 负责人 |
| --------------------------- | ----------------- | -------- | ----------------------------------------- | ------ |
| 多文件夹 → 保存工作区转化率 | V6.1 发布后 30 天 | >30%     | <30% 则调研原因，可能是保存流程复杂度过高 | 产品   |
| 工作区功能 NPS 评分         | V6.2 发布后 14 天 | >40 分   | <40 分则进行深度用户访谈                  | 产品   |

**定性研究计划**（V6.1 发布后 2 周内）:

- 访谈 5-8 位多文件夹功能的活跃用户
- 访谈 3-5 位未使用多文件夹的用户（了解原因）
- 输出：用户痛点报告，用于 V6.2 优先级调整

**样本量定义**:

- 多文件夹使用率：基于前 30 天活跃的独立设备数，至少 500+ 样本
- 首次打开时间：基于新用户首次启动，至少 100+ 样本
- 采样方式：全量统计（不抽样）

**决策流程**:

- <15% 使用率 → 产品负责人召集复盘会 → 48 小时内决策
- 决策选项：A) 继续优化 B) 降级为实验功能 C) 移除功能
