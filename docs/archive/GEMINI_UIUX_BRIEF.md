# 给 Gemini 的 UI/UX 交付简报

## 1. 项目背景

- 项目名：Writer
- 产品定位：本地 Markdown 写作工具
- 体验目标：Typora 风格（所见即所得、低干扰）+ Obsidian 基础目录管理

## 2. 当前阶段

- 阶段：Step C（UI/UX 风格冻结）
- 你要做的事：在既有功能边界内，给出更高审美的 UI/UX 方案，不扩功能范围。

## 3. 已冻结约束（必须遵守）

1. 平台：macOS + Windows（Linux 可选）
2. V1 仅实现 Must 12（不新增功能）
3. 单工作区（单根目录）
4. 自动保存：800ms 防抖 + 失焦 + 关闭前强制
5. 非目标：插件生态、双链/图谱、云同步协作、流程图、表格公式等

## 4. V1 Must 12（摘要）

1. 即时渲染编辑
2. 段落级格式快捷键
3. 行内样式快捷键
4. 代码块编辑
5. 图片粘贴与本地保存
6. 自动保存
7. 撤销/重做
8. 侧边栏文件树
9. 应用内新建文件
10. 文件重命名
11. 文件删除
12. 文件夹作为工作区

## 5. 当前基础文档

- `docs/frozen/PRD_V1.md`
- `docs/frozen/TECH_BASELINE.md`
- `docs/frozen/DESIGN_BASELINE.md`
- `docs/frozen/DECISIONS_LOG.md`

## 6. 你需要输出的交付物

请输出 1 份完整设计稿文档（Markdown），建议命名：`docs/archive/GEMINI_UIUX_PROPOSAL.md`，必须包含：

1. 设计方向（关键词 + 色彩 + 字体 + 版式）
2. 页面结构（信息架构图或清单）
3. 关键流程交互细化（打开工作区/新建/重命名/删除/粘贴图片/保存反馈）
4. 组件规范（侧栏、编辑区、状态栏、弹窗、内联错误）
5. 可访问性与可读性要点
6. 与 Must 12 的逐项映射（不能漏）
7. 不做项（明确不扩到 Should/Could）

## 7. 质量标准

1. 不能只给审美描述，必须可落到实现。
2. 所有建议必须能映射到 V1 Must。
3. 不得引入新功能范围。
4. 输出要能直接成为 Step C 复盘输入。

## 8. 评审口径（你可自检）

- 是否更好看且不增加认知负担？
- 是否保持“无打断写作”核心？
- 是否与技术基线冲突（Tauri + TipTap）？
