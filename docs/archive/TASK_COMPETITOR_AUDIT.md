# 任务说明：Typora + Obsidian 能力盘点（仅需求分析）

## 任务背景

当前项目处于原始需求分析阶段。为避免遗漏“记不清但关键好用”的能力，需要单独进行一次竞品能力盘点。

## 任务目标

输出一份可用于 V1 取舍的能力清单，帮助后续形成：

- V1 必须做
- 可延后
- 明确不做

注意：本任务只做需求分析，不进入架构设计与代码实现。

## 盘点对象与职责

- Typora：用于定义编辑体验基线（写作手感、输入行为、无打断流程）
- Obsidian：仅用于定义目录/文件管理基线（应用内新建、重命名、移动、切换）

## 范围（In Scope）

1. 编辑核心能力盘点（偏 Typora）
2. 文件树与文档管理能力盘点（偏 Obsidian）
3. 导出与基础文档流转能力盘点（PDF/HTML 等）
4. 高频使用场景梳理（写作主流程）
5. 候选功能的优先级建议（Must / Should / Could）

## 非范围（Out of Scope）

1. 插件生态
2. 双向链接、知识图谱、数据库化笔记
3. 云同步、多人协作、账号系统
4. 技术实现细节（框架选型、代码结构、接口设计）

## 强制边界规则

任何功能若不直接提升“无打断写作主流程”，默认不进入 V1。

## 输出格式（必须使用）

请输出一个表格，字段固定如下：

- 功能点
- 来源（Typora / Obsidian / 通用）
- 用户场景（1 句话）
- 使用频率判断（高/中/低）
- V1 建议（Must / Should / Could / Out）
- 理由（不超过 2 句）
- 验收口径（可观察、可测试）

并在表格后追加：

1. V1 功能清单（只列 Must）
2. V1 边界清单（明确 Out）
3. 待确认问题（最多 5 条）

## 完成定义（Definition of Done）

满足以下条件才算完成：

1. 功能清单覆盖编辑主流程与文件管理主流程
2. 每个候选项都有优先级与理由
3. 每个 Must 项都有可执行验收口径
4. 结论可以直接用于下一步 PRD 编写

## 质量要求

1. 不追求“大而全”，优先高频与关键路径
2. 不重复罗列同义功能
3. 不给实现方案，只给需求结论
4. 使用简洁中文，避免空泛描述

## 给执行模型的指令

你是需求分析助手，不是实现工程师。你必须优先保证边界清晰与可验收性，避免功能膨胀。
