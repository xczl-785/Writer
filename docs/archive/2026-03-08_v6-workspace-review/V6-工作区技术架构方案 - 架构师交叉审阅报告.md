## 2. Critical 问题汇总（阻塞性）

### 🔴 CR-01: FileWatcherService 技术选型根本性错误

**发现问题**:

方案中 FileWatcherService 使用 `chokidar`（Node.js 库）：

```typescript
import chokidar from 'chokidar';
this.watcher = chokidar.watch(workspaceFolders, { ... });
```

**问题严重性**:

- Tauri 前端运行在 WebView 环境，**无法使用 Node.js 原生模块**
- 即使能工作，前端处理大量文件事件也会造成 UI 阻塞
- 与 Tauri 安全模型冲突（前端不应直接访问文件系统）

**正确架构**:

```
Rust 后端（使用 notify crate）
    ↓ Tauri 事件推送
前端（监听 writer://file-change 事件）
```

**必须修改**: 完全重写 FileWatcherService，改为后端监听 + 前端订阅模式。

**审阅人一致建议**:

```rust
// Rust 后端实现
use notify::{Watcher, RecursiveMode, watcher};

#[tauri::command]
pub fn start_watching(
    app: tauri::AppHandle,
    paths: Vec<String>,
) -> Result<(), String> {
    let (tx, rx) = channel();
    let mut watcher = watcher(tx, Duration::from_millis(300))?;

    for path in paths {
        watcher.watch(path, RecursiveMode::Recursive)?;
    }

    // 后台线程转发事件到前端
    std::thread::spawn(move || {
        while let Ok(event) = rx.recv() {
            let _ = app.emit("writer://file-change", event);
        }
    });

    Ok(())
}
```

### 🔴 CR-02: 状态一致性缺乏事务保障机制

**发现问题**:

架构承诺 `workspaceActions` 提供"原子更新多个 store"，但 Zustand **没有 ACID 事务能力**。

以 `addFolderToWorkspace` 为例：

```typescript
addFolderToWorkspace: async (folderPath: string): Promise<Result> => {
  // 2. 加载文件树（批量）← 这里可能失败
  // 3. 原子更新 workspaceSlice + filetreeSlice ← 这不是原子的！
  // 4. 标记 isDirty = true
};
```

**风险场景**:

1. 加载文件树成功
2. 更新 `workspaceSlice.folders` 成功
3. 更新 `filetreeSlice.rootFolders` 失败
4. **结果**: 两个 store 状态不一致

**必须修改**: 添加快照 - 回滚机制：

```typescript
addFolderToWorkspace: async (folderPath: string): Promise<Result> => {
  // 1. 预先验证和准备数据
  const treeResult = await FsService.listTree(folderPath);
  if (!treeResult.ok) return { ok: false, error: treeResult.error };

  // 2. 快照当前状态（用于回滚）
  const snapshot = {
    folders: useWorkspaceStore.getState().folders,
    rootFolders: useFileTreeStore.getState().rootFolders,
  };

  try {
    // 3. 批量更新
    useWorkspaceStore.getState().addFolder(newFolder);
    useFileTreeStore.getState().addRootFolder(newRoot);

    // 4. 验证状态一致性
    assertConsistency();
  } catch (error) {
    // 5. 回滚
    useWorkspaceStore.setState({ folders: snapshot.folders });
    useFileTreeStore.setState({ rootFolders: snapshot.rootFolders });
    throw error;
  }
};
```

## 3. Medium 问题汇总

### 🟡 M-01: 路径安全验证完全缺失

**后端工程师指出**:

- 当前设计没有任何路径验证逻辑
- 存在路径穿越风险（`../../etc/passwd`）
- 缺少工作区边界检查

**必须补充**:

```rust
pub struct WorkspaceAllowlist {
    roots: Vec<PathBuf>,
}

impl WorkspaceAllowlist {
    pub fn validate_path(&self, path: &str) -> Result<PathBuf, String> {
        let canonical = PathBuf::from(path).canonicalize()?;

        if !self.roots.iter().any(|root| canonical.starts_with(root)) {
            return Err("Path outside workspace boundaries".to_string());
        }
        Ok(canonical)
    }
}
```

### 🟡 M-02: list_tree_batch 串行执行浪费并发能力

**问题**:

```rust
// 方案中的串行实现
paths.into_iter().map(|path| list_tree(&path)...).collect()
```

**建议改进**:

```rust
// 并行实现
use std::thread;

paths.into_iter()
    .map(|path| {
        thread::spawn(move || list_tree(&path)...)
    })
    .collect::<Vec<_>>()
    .into_iter()
    .map(|h| h.join().unwrap())
    .collect()
```

### 🟡 M-04: 原子写入安全性不足

**当前设计**:

```rust
let temp_path = format!("{}.tmp", path);
fs::write(&temp_path, &content)?;
fs::rename(&temp_path, path)?;
```

**问题**:

- 临时文件名可预测，存在竞态条件
- `rename` 失败后临时文件残留
- 跨文件系统 rename 会失败

**建议使用 tempfile crate**:

```rust
use tempfile::NamedTempFile;

let mut temp_file = NamedTempFile::new_in(parent)?;
temp_file.write_all(content.as_bytes())?;
temp_file.persist(target_path)?;
```
