# AI 辅助开发中的唯一真源构建方案

> 解决问题：在使用 AI 开发过程中，如何确保逻辑的一致性和可追溯性，特别是在 MVP → 迭代开发过程中，新旧需求纠缠导致的逻辑混乱问题。

---

## 一、问题背景

### 1.1 典型案例

```
阶段1 (MVP)
├── 需求：侧边栏添加"新建文件"、"新建文件夹"按钮
├── 实现：按钮 + 路径判断 + 禁用/启用逻辑
└── 问题：所有逻辑内嵌在侧边栏组件中，未记录决策

阶段2 (迭代)
├── 需求：顶部菜单栏也添加这两个按钮
├── 实现：AI 新对话，无上下文，重新实现了一套逻辑
└── 结果：两套行为不一致，用户困惑
```

### 1.2 根因分析

| 层面 | 问题描述 |
|------|----------|
| 知识存储 | 决策散落在代码、对话中，隐式且碎片化 |
| 上下文丢失 | 每次新对话 AI 从零开始，不记得之前的约束 |
| 一致性校验 | 新需求覆盖旧逻辑时，无检查机制 |
| 人的角色 | 人只做决策，但决策未被记录和追溯 |

### 1.3 核心矛盾

```
错误的心智模型：
"侧边栏的新建按钮"是一个功能

正确的心智模型：
"新建文件"是一个能力
  └─ 可从多个入口触发（侧边栏、顶部菜单、右键菜单、快捷键）
  └─ 所有入口共享同一套逻辑
```

## 二、唯一真源体系架构

### 2.1 三层结构

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 1: 决策层 (Decision Layer)                           │
│  ─────────────────────────────────────────────────────────  │
│  • 能力目录 (Capability Registry)                           │
│  • 决策日志 (Decision Log)                                  │
│  • 约束清单 (Constraint List)                               │
│  • 影响地图 (Impact Map)                                    │
│                                                             │
│  作用：人做决策的记录，AI 开发时的"宪法"                      │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ 指导
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Layer 2: 能力层 (Capability Layer)                         │
│  ─────────────────────────────────────────────────────────  │
│  • 行为定义（做什么）                                        │
│  • 前置条件（何时可用）                                      │
│  • 执行逻辑（如何做）                                        │
│  • 一致性约束（不变量）                                      │
│                                                             │
│  作用：与 UI 解耦的纯逻辑单元                                 │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ 实现
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Layer 3: 表现层 (Presentation Layer)                        │
│  ─────────────────────────────────────────────────────────  │
│  • UI 组件（侧边栏、顶部菜单、右键菜单）                       │
│  • 触发入口                                                  │
│                                                             │
│  作用：只做"触发"，不做"判断"                                 │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 核心原则

1. **行为与 UI 解耦** — 能力独立于入口存在
2. **决策点前置** — 先定义能力，再实现入口
3. **AI 主动查询** — 新需求进入时先查能力目录
4. **人确认影响** — 发现相关能力时暂停让人决策

## 三、需要维护的内容

### 3.1 能力目录 (Capability Registry)

**定义**：项目所有"能力"的清单，记录有哪些能力、从哪些入口触发、依赖关系。

**目的**：
- AI 快速定位是否有可复用的能力
- 防止重复创建相似功能
- 维护入口一致性

**目录结构**：
```
/capabilities/
  ├── file-operations/
  │   ├── create-file.md
  │   ├── create-folder.md
  │   ├── delete-item.md
  │   └── rename-item.md
  ├── search/
  │   ├── search-file.md
  │   └── search-content.md
  └── export/
      └── export-project.md
```

**文档模板**：
```markdown
# [capability-id]

## 元信息
- 创建日期：YYYY-MM-DD
- 最后更新：YYYY-MM-DD
- 状态：[进行中 | 稳定 | 废弃]

## 概述
[能力的一句话描述]

## 触发入口
- [x] 侧边栏按钮
- [x] 顶部菜单
- [ ] 右键菜单
- [ ] 快捷键
- [ ] 命令面板

## 前置条件
1. [条件1：如当前路径有效]
2. [条件2：如用户有写权限]
3. [条件3：如不在只读模式]

## 行为定义
1. [步骤1]
2. [步骤2]
3. [步骤3]

## 状态计算
- 启用条件：[描述]
- 禁用条件：[描述]

## 一致性约束
- [约束1：所有触发入口必须满足相同的前置条件]
- [约束2：禁用状态计算逻辑统一]

## 依赖
- 依赖能力：[其他 capability-id]
- 被依赖：[其他 capability-id]

## 相关决策
- [决策日志链接]

## 备注
[其他需要记录的信息]
```

### 3.2 决策日志 (Decision Log)

**定义**：重要决策的 why/what/when/who 记录。

**目的**：
- 追溯决策来源
- 理解约束条件的历史原因
- 避免重复讨论已决策的问题

**目录结构**：
```
/decisions/
  ├── 2024-01-15-sidebar-file-buttons.md
  ├── 2024-02-20-topbar-menu-buttons.md
  └── 2024-03-10-export-feature.md
```

**文档模板**：
```markdown
# [YYYY-MM-DD] [决策标题]

## 元信息
- 日期：YYYY-MM-DD
- 决策者：[人名/AI辅助]
- 阶段：[MVP | 迭代版本号]
- 相关需求：[需求ID或描述]

## 背景
[为什么需要做这个决策？当前面临的问题是什么？]

## 决策内容
[具体做了什么决策？]

## 约束条件
[决策带来的约束或限制]

## 备选方案
### 方案A：[名称]
- 描述：[简述]
- 是否采纳：[是/否]
- 理由：[为什么采纳/否决]

### 方案B：[名称]
- 描述：[简述]
- 是否采纳：[是/否]
- 理由：[为什么采纳/否决]

## 影响范围
- 影响的能力：[capability-id 列表]
- 影响的模块：[模块名称列表]
- 后续需注意：[给未来开发的提示]

## 相关文档
- 能力定义：[链接]
- 代码位置：[链接]
```

### 3.3 约束清单 (Constraint List)

**定义**：项目中的业务规则、技术约束、一致性要求。

**目的**：
- AI 开发时知道哪些是不可违反的
- 新功能设计时快速了解边界

**目录结构**：
```
/constraints/
  ├── business-rules.md
  ├── technical-limits.md
  └── consistency-requirements.md
```

**文档模板**：
```markdown
# [约束类型名称]

## 概述
[约束的总体描述]

## 约束列表

### [约束ID-001] [约束名称]
- 类型：[业务规则 | 技术约束 | 一致性要求]
- 级别：[必须 | 建议]
- 适用范围：[影响的模块/能力]
- 描述：[详细描述]
- 来源：[决策日志链接]
- 示例：[具体例子]

### [约束ID-002] [约束名称]
...
```

**示例**：
```markdown
# 文件操作约束

## 约束列表

### FILE-001 路径有效性校验
- 类型：业务规则
- 级别：必须
- 适用范围：create-file, create-folder, delete-item, rename-item
- 描述：所有文件操作必须先校验当前路径是否有效，无效时禁用操作入口
- 来源：decisions/2024-01-15-sidebar-file-buttons.md
- 示例：用户在根目录时，新建文件按钮应禁用

### FILE-002 操作入口一致性
- 类型：一致性要求
- 级别：必须
- 适用范围：所有文件操作能力
- 描述：同一能力的所有触发入口必须共享相同的启用/禁用逻辑
- 来源：decisions/2024-02-20-topbar-menu-buttons.md
- 示例：侧边栏和顶部菜单的"新建文件"按钮必须同时启用或禁用
```

### 3.4 影响地图 (Impact Map)

**定义**：能力、模块、入口之间的映射关系。

**目的**：
- 快速查询修改某处会影响哪些地方
- 新需求进入时快速定位相关能力

**目录结构**：
```
/impact-map/
  ├── capability-to-entries.md    # 能力 → 入口映射
  ├── module-dependencies.md       # 模块依赖关系
  └── change-history.md            # 变更历史
```

**文档模板**：

**capability-to-entries.md**：
```markdown
# 能力入口映射

## 能力索引

| 能力ID | 能力名称 | 触发入口 | 相关约束 |
|--------|----------|----------|----------|
| create-file | 新建文件 | 侧边栏、顶部菜单 | FILE-001, FILE-002 |
| create-folder | 新建文件夹 | 侧边栏、顶部菜单 | FILE-001, FILE-002 |
| delete-item | 删除项目 | 侧边栏、右键菜单 | FILE-001 |
| rename-item | 重命名 | 侧边栏、右键菜单 | FILE-001 |

## 入口索引

| 入口 | 关联能力 |
|------|----------|
| 侧边栏 | create-file, create-folder, delete-item, rename-item |
| 顶部菜单 | create-file, create-folder |
| 右键菜单 | delete-item, rename-item |
```

**module-dependencies.md**：
```markdown
# 模块依赖关系

## 依赖图

```
[sidebar] ──依赖──▶ [file-operations]
    │
[topbar] ──依赖──▶ [file-operations]
    │
[context-menu] ──依赖──▶ [file-operations]
```

## 详细依赖

### [module-name]
- 依赖模块：[列表]
- 被依赖：[列表]
- 包含能力：[capability-id 列表]
```

## 四、更新时机与调用流程

### 4.1 更新时机总览

| 文档 | 创建时机 | 更新时机 | 触发者 |
|------|----------|----------|--------|
| 能力目录 | 新能力定义时 | 入口变更、行为变更时 | 人确认后 AI 更新 |
| 决策日志 | 重要决策做出时 | 决策被修正时 | 人确认后 AI 更新 |
| 约束清单 | 发现新约束时 | 约束变更时 | 人确认后 AI 更新 |
| 影响地图 | 能力/模块变更时 | 关系变化时 | AI 自动更新 |

### 4.2 调用流程

```
┌─────────────────────────────────────────────────────────────┐
│                      新需求进入                              │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 1: AI 读取能力目录 + 影响地图                          │
│  ─────────────────────────────────────────────────────────  │
│  问题：是否涉及已有能力？                                    │
└─────────────────────────────────────────────────────────────┘
                              │
              ┌───────────────┴───────────────┐
              │                               │
             是                               否
              │                               │
              ▼                               ▼
┌─────────────────────────┐       ┌─────────────────────────┐
│  Step 2a: AI 提示人      │       │  Step 2b: 创建新能力    │
│  "发现相关能力：         │       │  ─────────────────────  │
│   - capability-A        │       │  • 更新能力目录          │
│   - capability-B"       │       │  • 记录决策日志          │
│                         │       │  • 定义约束条件          │
│  是否复用？是否需要      │       │  • 更新影响地图          │

              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 3: 人做决策                                           │
│  ─────────────────────────────────────────────────────────  │
│  • 复用已有能力？仅添加新入口？                               │
│  • 调整已有能力？需要更新哪些约束？                           │
│  • 新建独立能力？与已有能力的关系？                           │
└─────────────────────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 4: 更新文档                                           │
│  ─────────────────────────────────────────────────────────  │
│  • 更新能力目录（新入口/行为变更）                            │
│  • 更新决策日志（记录本次决策）                               │
│  • 更新约束清单（如有新约束）                                 │
│  • 更新影响地图（关系变化）                                   │
└─────────────────────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 5: 实现代码                                           │
│  ─────────────────────────────────────────────────────────  │
│  • AI 按照能力定义编写代码                                    │
│  • 人审核代码是否符合约束                                     │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 人的角色分工

| 环节 | AI 的职责 | 人的职责 |
|------|-----------|----------|
| 需求分析 | 搜索能力目录，发现相关能力 | 理解影响，做决策 |
| 方案设计 | 提示可选方案，标注影响范围 | 选择方案，确认约束 |
| 文档维护 | 生成文档草稿 | 审核补充，确认 why |
| 代码实现 | 按能力定义编写代码 | 审核，验收 |
| 一致性检查 | 提示"可能需要一致性检查" | 决定是否强制执行 |

## 五、防止问题的机制

### 5.1 强制检查点

```
┌─────────────────────────────────────────────────────────────┐
│  Checkpoint 1: 能力搜索                                     │
│  ─────────────────────────────────────────────────────────  │
│  触发：新需求进入                                            │
│  动作：AI 必须读取能力目录，搜索关键词                        │
│  输出：相关能力列表 / 无相关能力                              │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Checkpoint 2: 影响确认                                      │
│  ─────────────────────────────────────────────────────────  │
│  触发：发现相关能力时                                        │
│  动作：AI 提示人，暂停等待决策                                │
│  输出：人的确认（复用/调整/新建）                             │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Checkpoint 3: 约束校验                                      │
│  ─────────────────────────────────────────────────────────  │
│  触发：设计实现方案时                                        │
│  动作：AI 检查是否违反约束清单                                │
│  输出：合规 / 违规提示                                       │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Checkpoint 4: 文档更新                                      │
│  ─────────────────────────────────────────────────────────  │
│  触发：实现完成后                                            │
│  动作：AI 更新相关文档                                       │
│  输出：更新后的能力目录/决策日志/影响地图                     │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 防呆设计

| 场景 | 防护措施 |
|------|----------|
| AI 不知道要查能力目录 | 在项目 AGENTS.md 或 README 中写明"开发前必读" |
| AI 静默创建重复逻辑 | 约束清单中写明"新增入口必须复用已有能力" |
| 人忘记记录决策 | AI 实现后自动生成决策日志草稿，提醒人确认 |
| 文档与代码不一致 | 定期（如每个版本）进行文档-代码一致性审查 |

## 六、项目初始化模板

### 6.1 目录结构

```
project/
├── .ai-context/                    # AI 上下文目录
│   ├── README.md                   # 给 AI 的指引
│   ├── capabilities/               # 能力目录
│   │   ├── index.md                # 能力索引
│   │   └── [category]/             # 分类目录
│   │       └── [capability-id].md
│   ├── decisions/                  # 决策日志
│   │   └── [YYYY-MM-DD-title].md
│   ├── constraints/                # 约束清单
│   │   └── [constraint-type].md
│   └── impact-map/                 # 影响地图
│       ├── capability-to-entries.md
│       └── module-dependencies.md
├── src/                            # 源代码
└── [其他项目文件]
```

### 6.2 AI 指引文件 (.ai-context/README.md)

```markdown
# AI 开发指引

## 开发前必读

在开始任何开发工作前，请按以下顺序阅读：

1. `/capabilities/index.md` - 了解项目有哪些能力
2. `/constraints/` - 了解项目的约束条件
3. `/impact-map/` - 了解模块依赖关系

## 开发流程

1. **搜索**：新需求进入时，先搜索能力目录
2. **确认**：发现相关能力时，提示用户确认是否复用
3. **决策**：记录用户的决策到决策日志
4. **实现**：按照能力定义编写代码
5. **更新**：更新能力目录、影响地图

## 文档格式

所有文档使用 Markdown 格式，便于 AI 解析和人类阅读。

## 命名规范

- 能力ID：使用 kebab-case，如 `create-file`
- 决策日志：`YYYY-MM-DD-title.md`
- 约束ID：使用大写 + 序号，如 `FILE-001`
```

## 七、最佳实践

### 7.1 能力拆分原则

```
正确的拆分：
├── create-file          # 一个能力
├── create-folder        # 一个能力
└── delete-item          # 一个能力

错误的拆分：
├── sidebar-create-file  # ❌ 按位置拆分
├── topbar-create-file   # ❌ 同一行为不应拆成多个能力
└── create-file-folder   # ❌ 粒度太大
```

### 7.2 决策记录原则

```
好的决策记录：
- 记录 why（为什么做这个决策）
- 记录备选方案及否决理由
- 记录影响范围
- 记录约束条件

不好的决策记录：
- 只记录 what（做了什么）
- 没有上下文
- 没有追溯价值
```

### 7.3 约束定义原则

```
好的约束：
- 具体可验证
- 有来源可追溯
- 有适用范围

不好的约束：
- 模糊（如"保持一致性"）
- 无来源
- 范围不明确
```
