# Writer 通用错误提示分级交互规范 (Design Doc)

## 1. 核心设计理念 (Core Principles)

基于 Writer "无打断写作主流程优先" 的产品核心原则，错误提示的交互设计摒弃了"一刀切的全局弹窗"或"统一推送到状态栏"的方案。

我们将错误按照**"对用户当前输入内容及数据安全的影响程度（Severity/Impact）"**进行分级。不同级别的错误将映射到三种不同侵入度（Intrusiveness）的 UI 容器中，确保在日常沉浸写作中**零打扰**，而在面临数据丢失风险时**强阻断**。

---

## 2. 错误分级与 UI 映射策略 (The 3-Level Strategy)

### Level 1: 隐性轻量提醒 (Silent Info / Minor Error)

- **适用场景**：非关键性的后台失败，如"自动保存失败（但本地已缓存）"、"云同步短暂掉线"、"某非核心插件加载失败"。
- **UI 容器**：**底部状态栏指示器 (Status Bar Indicator)**
- **交互表现**：
  - **零弹窗、零动画打扰**。
  - 底部状态指示灯从正常的绿色平滑过渡为橙色（警告）或红色（错误）。
  - 状态文本变更为"未保存"或"离线"。
- **操作 (Action)**：点击状态栏指示器，可展开一个小型 Popover/Tooltip 显示具体 `reason` 并提供"重试"按钮。

### Level 2: 显性操作反馈 (Transient Warning)

- **适用场景**：用户主动触发但失败的操作（如"导出 PDF 失败"），或需要用户知晓但无数据丢失风险的系统错误。
- **UI 容器**：**顶部中央轻提示 (Top-Center Toast)**
- **交互表现**：
  - 在主编辑区顶部正中央，从上向下滑出一个带模糊背景（backdrop-blur）的精致胶囊状 Toast。
  - 包含图标、简短描述文字、以及可选的操作文本按钮。
- **生命周期**：出现后停留 3~5 秒，随后伴随平滑动画自动向上收起并消失。
- **位置优势**：视觉动线短，紧贴主视区但通常不遮挡实际输入的光标位置（光标多在屏幕中下部）。

### Level 3: 致命阻断横幅 (Critical Banner) — 覆盖层方案 (Overlay Banner, Option Y)

- **适用场景**：极度危险的错误，继续输入大概率导致数据丢失。例如："当前编辑的文件已被外部程序意外删除"、"当前文件失去磁盘写入权限变成只读"、"磁盘空间已满"。
- **UI 容器**：**编辑器覆盖横幅 (Overlay Banner)**
- **交互表现**：
  - 在编辑器视图顶部悬浮显示一条全宽的警示横幅，**不挤压正文内容**，而是以覆盖层形式悬停于编辑器区域上方。
  - 采用柔和的红色或橙色背景（如 `bg-red-50`配合`border-red-200`），文字说明原因。
  - 可选添加轻微的阴影效果（如 `shadow-lg`）以增强层次感。
- **生命周期**：**持久存在，直到问题被解决**。
- **强阻断性**：横幅以悬浮覆盖形式呈现，视觉上"叠加"于编辑器内容之上，确保用户无法忽视。用户必须点击横幅提供的 Action（如"重新另存为"或"放弃修改"）来解除警报。这确保了在极度危险情况下的防错优先级，同时避免了对编辑器布局的破坏。

---

## 3. 架构适配性 (Architecture Alignment)

此设计解耦了现有的具体错误分类（`permission`, `network`, `user`, `system`），而是依赖于前端 `ErrorService` 抛出错误时应附加的"严重程度（Severity）"属性。
这意味着未来无论新增何种类型的错误，只需评估其对数据安全的影响，即可自动落入对应的三个 UI 层级之一，降低了系统演进的心智压力。

---

## 4. 原型演示说明 (Prototype Demo)

为了直观展示这三种效果，对应的 HTML 原型 (`prototype_v6_windows_error_ui.html`) 提供了一个悬浮的"Debug 面板"，允许用户主动点击按钮来模拟触发这三个层级的错误表现。
