## 三、状态管理架构

### 3.1 架构模式

采用 **领域分层 + 多 Zustand Store 模式**：

- 每个领域维护自己的 store / service / UI 边界
- 应用级运行时状态保留在 `state/slices/`
- 跨领域协调收敛到 `domains/*/services/`，例如 `workspaceActions.ts` 与 `fileActions.ts`
- `src/i18n/`、`src/utils/` 中保留少量兼容导出，供历史入口与测试复用

### 3.2 Store 职责

| Store               | 位置 | 职责           | 关键状态                                    |
| ------------------- | ---- | -------------- | ------------------------------------------- |
| `useEditorStore`    | domains/editor/state | 多文档内容管理 | `fileStates: Record<path, FileEditorState>` |
| `useFileTreeStore`  | domains/file/state | 文件树 UI 状态 | `rootFolders`, `expandedPaths`, `selectedPath` |
| `useWorkspaceStore` | domains/workspace/state | 工作区文件管理 | `folders`, `workspaceFile`, `openFiles`, `activeFile` |
| `useStatusStore`    | state/slices | 应用状态通知   | `status`, `saveStatus`, `saveError`         |
| `useSettingsStore`  | domains/settings/state | 用户偏好持久化 | `localePreference`, `typewriterEnabledByUser`, `focusZenEnabledByUser` |
| `useViewModeStore`  | state/slices | 运行时视图态   | `isZen`, `isFocusZen`, `isTypewriterActive` |

### 3.3 数据流

```
用户操作 → Component → Domain Service / Store Action → store.set() → State Update → Component Re-render
```

**示例：打开文件流程**

```
用户点击文件
    │
    ▼
workspaceActions.openFile(path)
    │
    ├── flushAffectedFiles(activeFile)  // 保存当前文件
    ├── FsService.readFile(path)        // 读取目标文件
    ├── useEditorStore.initializeFile() // 更新 editor store
    └── useWorkspaceStore.openFile()    // 更新 workspace store
    │
    ▼
订阅组件自动重新渲染
```

## 四、组件架构

### 4.1 组件层级

```
App
├── Sidebar
├── Editor
│   └── domains/editor/core/Editor
│       └── EditorOrchestrator → EditorImpl → EditorView → EditorShell
├── StatusBar
├── SettingsPanel
└── RecentWorkspacesMenu / EmptyStateWorkspace
```

### 4.2 组件通信模式

- **Props 传递**: 父子组件通信（主要模式）
- **自定义 Hook**: 逻辑复用（useOutlineExtractor, useContextMenu）
- **事件驱动**: 跨组件通信（editor.update → Outline 刷新）
- **注册中心**: ContextMenuRegistry（全局菜单管理）
- **Portal 渲染**: ContextMenu 脱离 DOM 层级

对于文件树创建相关交互，当前采用“**菜单命令 / Sidebar 事件 / Explorer 命令**”三段式桥接：

- 顶部菜单通过 `menuCommandBus` 进入 `fileCommands.ts`
- `createEntryCommands.ts` 负责把菜单创建命令映射为 Sidebar 创建目标，并处理 Sidebar 显示时机
- `Sidebar.tsx` 再通过 `resolveCreateEntryExplorerCommand()` 汇聚到 `dispatchExplorerCommand()`
- 侧边栏按钮、空白区右键菜单与顶部菜单最终共享同一套工作区可用性语义

### 4.3 编辑区分层（领域驱动）

- `domains/editor/core/`: 布局模型与编排入口（`EditorLayoutModel`, `EditorOrchestrator`）
- `domains/editor/domain/`: 纯逻辑（find/typewriter/focus-zen 等）
- `domains/editor/integration/`: 菜单/右键/粘贴/持久化桥接
- `domains/editor/view/`: 渲染边界（`EditorView`）
- `domains/editor/ui/`: 具体 UI 组件（`EditorShell` 等）
- `shared/components/`: 跨领域复用组件（`ContextMenu`、`Dialog`、`Outline`）

## 五、关键设计决策

### 5.1 为什么选择领域驱动架构？

- **高内聚**: 相关代码组织在同一领域目录下
- **低耦合**: 领域间通过明确定义的接口通信
- **可维护性**: 新功能有明确的归属位置
- **可测试性**: 领域逻辑可独立测试

### 5.2 为什么选择 Zustand？

- 轻量级（比 Redux 简洁）
- 无需 Provider 包裹
- 支持在非组件环境使用（通过 `getState()`）
- TypeScript 友好

### 5.3 服务层设计原则

- 服务优先收敛到所属领域目录，跨领域共享能力保留在 `src/services/` 与 `src/shared/`
- `workspaceActions`、`fileActions` 是当前跨 store 协调的真源入口
- `createEntryCommands` 负责“新建文件 / 新建文件夹”入口的语义收敛，不承载具体文件系统创建逻辑
- FsService 作为文件系统与配置读写的后端调用入口
- ErrorService 统一错误处理

### 5.4 本轮创建入口重构后的边界

本轮小重构明确了以下边界：

- `fileCommands.ts`：负责菜单命令注册，不再直接维护 `new-file` / `new-folder` 的延迟派发细节
- `createEntryCommands.ts`：负责创建入口映射、Sidebar 展示时机、工作区可用性判定
- `Sidebar.tsx`：负责展示层与 Explorer 命令调度，不再散落维护创建命令字符串映射
- `menuSchema.ts`：通过 `FILE_MENU_CREATE_ITEMS` 统一维护文件菜单中的创建项定义
