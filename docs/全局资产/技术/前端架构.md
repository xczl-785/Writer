## 三、状态管理架构

### 3.1 架构模式

采用 **多 Zustand Store 模式**：

- 每个 slice 都是独立的 store
- Slice 内包含 state + actions
- 跨 slice 协调通过 `actions/` 目录实现

### 3.2 Store 职责

| Store               | 职责           | 关键状态                                    |
| ------------------- | -------------- | ------------------------------------------- |
| `useEditorStore`    | 多文档内容管理 | `fileStates: Record<path, FileEditorState>` |
| `useFileTreeStore`  | 文件树 UI 状态 | `nodes`, `expandedPaths`, `selectedPath`    |
| `useWorkspaceStore` | 工作区文件管理 | `currentPath`, `openFiles`, `activeFile`    |
| `useStatusStore`    | 应用状态通知   | `status`, `saveStatus`, `saveError`         |
| `useSettingsStore`  | 用户偏好持久化 | `localePreference`, `typewriterEnabledByUser`, `focusZenEnabledByUser` |
| `useViewModeStore`  | 运行时视图态   | `isZen`, `isFocusZen`, `isTypewriterActive` |

### 3.3 数据流

```
用户操作 → Component → Action Function → store.set() → State Update → Component Re-render
```

**示例：打开文件流程**

```
用户点击文件
    │
    ▼
workspaceActions.openFile(path)
    │
    ├── flushAffectedFiles(activeFile)  // 保存当前文件
    ├── FsService.readFile(path)        // 读取目标文件
    ├── useEditorStore.initializeFile() // 更新 editor store
    └── useWorkspaceStore.openFile()    // 更新 workspace store
    │
    ▼
订阅组件自动重新渲染
```

## 四、组件架构

### 4.1 组件层级

```
App
├── Sidebar
│   ├── FileTree
│   └── SidebarHeader
├── Editor
│   ├── EditorOrchestrator (core/)
│   ├── EditorImpl
│   └── EditorView → EditorShell
│       ├── Toolbar
│       ├── FindReplacePanel
│       ├── Outline
│       ├── SlashMenu
│       ├── BubbleMenu
│       └── GhostHint
└── StatusBar
```

### 4.2 组件通信模式

- **Props 传递**: 父子组件通信（主要模式）
- **自定义 Hook**: 逻辑复用（useOutlineExtractor, useContextMenu）
- **事件驱动**: 跨组件通信（editor.update → Outline 刷新）
- **注册中心**: ContextMenuRegistry（全局菜单管理）
- **Portal 渲染**: ContextMenu 脱离 DOM 层级

### 4.3 编辑区分层（当前）

- `core/`: 布局模型与编排入口（`EditorLayoutModel`, `EditorOrchestrator`）
- `domain/`: 纯逻辑（find/typewriter/focus-zen 等）
- `integration/`: 菜单/右键/粘贴/持久化桥接
- `view/`: 渲染边界（`EditorView`）
- `components/`: 具体 UI 组件（`EditorShell` 等）

## 五、关键设计决策

### 5.1 为什么选择 Zustand？

- 轻量级（比 Redux 简洁）
- 无需 Provider 包裹
- 支持在非组件环境使用（通过 `getState()`）
- TypeScript 友好

### 5.2 为什么使用多 Store 模式？

- 每个 Store 职责单一，易于理解
- 避免单一 Store 过于庞大
- 细粒度更新，性能更优
- 便于测试和维护

### 5.3 服务层设计原则

- 服务使用对象字面量实现（无实例化）
- FsService 作为后端调用唯一入口
- ErrorService 统一错误处理
