# 标题栏与菜单栏改造优化清单

更新时间：2026-03-14

## 背景

本清单用于记录本轮 `Windows/macOS` 标题栏与菜单栏改造后，已确认可运行但实现上仍不够优雅的部分，供下一窗口继续收口。

本清单只记录工程优化项，不调整需求定义。

## 优化项

### 1. 侧边栏按钮行为在双平台重复实现

- 当前 [WindowsTitleBar.tsx](/E:/Project/Writer/src/ui/chrome/WindowsTitleBar.tsx) 和 [MacTitleBar.tsx](/E:/Project/Writer/src/ui/chrome/MacTitleBar.tsx) 各自维护了一套：
  - `sidebarClickTimerRef`
  - 单击延时判定
  - 双击切换极致禅模式
  - 极致禅模式下单击先退出
- 风险：
  - 后续如果再调整交互节奏或行为规则，需要改两份，容易漂移。
- 建议：
  - 抽出共享 hook，例如 `useSidebarToggleBehavior`。

### 2. App 到标题栏的状态透传开始膨胀

- 当前 [App.tsx](/E:/Project/Writer/src/app/App.tsx) 需要向 [PlatformTitleBar.tsx](/E:/Project/Writer/src/ui/chrome/PlatformTitleBar.tsx) 传递：
  - `isSidebarVisible`
  - `isFocusZen`
  - `isHeaderAwake`
  - `onToggleSidebar`
  - `onSetFocusZen`
- 风险：
  - 标题栏继续演进后，`App` 会逐步变成大量 UI 控制状态的转发层。
- 建议：
  - 抽出 `AppChrome` 容器或 `chromeState/chromeActions` 边界。

### 3. Zen / Focus Zen 状态模型不够清晰

- 当前 [viewModeSlice.ts](/E:/Project/Writer/src/state/slices/viewModeSlice.ts) 中同时存在：
  - `isZen`
  - `isFocusZen`
  - `isTypewriterActive`
  - `typewriterRestoreSnapshot`
- 当前 [App.tsx](/E:/Project/Writer/src/app/App.tsx) 中 `toggleSidebar`、`applyFocusZen`、`enterZen`、`exitZen` 之间存在隐式耦合。
- 风险：
  - “收起侧边栏”“普通禅模式”“极致禅模式”容易再次串味，后续再改一个入口就可能引入回归。
- 建议：
  - 将视图模式改为更显式的状态机或单一来源状态模型。

### 4. WindowsTitleBar 组件职责过重

- 当前 [WindowsTitleBar.tsx](/E:/Project/Writer/src/ui/chrome/WindowsTitleBar.tsx) 同时负责：
  - 窗口状态监听
  - 标题栏拖拽
  - 品牌区渲染
  - 菜单栏渲染
  - 侧边栏按钮行为
  - 窗口控制按钮
  - 极致禅模式隐藏
- 风险：
  - 单组件职责过多，定位交互问题和继续迭代成本都会升高。
- 建议：
  - 拆出子组件或 hook，例如：
    - `WindowControls`
    - `BrandStrip`
    - `TitleDragSurface`
    - `SidebarToggleButton`
    - `useWindowChromeState`

### 5. 编辑器边界仍保留了迁移后的遗留接口

- 当前 [editorTypes.ts](/E:/Project/Writer/src/domains/editor/core/editorTypes.ts) 里仍保留：
  - `isSidebarVisible?: boolean`
  - `onToggleSidebar?: () => void`
- 但编辑器内部入口已经迁移到标题栏，这组参数当前不再承担原本职责。
- 风险：
  - 会让后续读代码的人误以为编辑器仍然控制侧边栏入口。
- 建议：
  - 下一轮确认没有其他调用依赖后，清理这组遗留接口。

### 6. 当前回归测试偏向源码字符串断言

- 当前新增测试主要集中在：
  - [PlatformTitleBarIntegration.test.ts](/E:/Project/Writer/src/app/PlatformTitleBarIntegration.test.ts)
  - [WindowsTitleBarIntegration.test.ts](/E:/Project/Writer/src/ui/chrome/WindowsTitleBarIntegration.test.ts)
  - [SidebarResponsiveBehavior.test.ts](/E:/Project/Writer/src/ui/sidebar/SidebarResponsiveBehavior.test.ts)
- 这些测试能防止明显回退，但更多是在约束源码结构和字面量。
- 风险：
  - 对重构不友好，也不能完全证明交互行为本身正确。
- 建议：
  - 增加真实交互测试：
    - 单击标题栏侧边栏按钮
    - 双击标题栏侧边栏按钮
    - 极致禅模式下标题栏显隐
    - 菜单栏显隐
    - 窗口按钮行为

### 7. 极致禅模式的“头部”控制策略仍然分散

- 当前头部相关 UI 分散在两处：
  - 编辑器头部：[EditorShell.tsx](/E:/Project/Writer/src/domains/editor/ui/components/EditorShell.tsx)
  - 平台标题栏：[WindowsTitleBar.tsx](/E:/Project/Writer/src/ui/chrome/WindowsTitleBar.tsx)、[MacTitleBar.tsx](/E:/Project/Writer/src/ui/chrome/MacTitleBar.tsx)
- 目前已经通过 `isHeaderAwake` 接上，但策略仍然是多点分布。
- 风险：
  - 后续只改一侧时，容易再出现“编辑器头隐藏了，但标题栏没隐藏”这类回归。
- 建议：
  - 抽象统一的 `header visibility policy`，减少多处各自判断。

## 下一窗口建议优先级

1. 抽出共享的侧边栏按钮行为 hook，消除 Win/Mac 重复实现。
2. 收口 `Zen / Focus Zen` 状态模型，理顺“收起侧边栏”和“进入极致禅模式”的边界。
3. 把标题栏与极致禅模式的显隐逻辑集中管理。
4. 用真实交互测试替换一部分源码字符串测试。
