## 2. 文档目标

本文档用于基于概念定义与需求定义，描述用户在工作区相关任务中的关键交互场景，明确：

1. 前置状态是什么；
2. 用户执行什么动作；
3. 系统应如何响应；
4. 最终应进入什么状态；
5. 当前场景中的菜单与状态栏语义应如何成立。

---

## 3. 场景定义原则

### 3.1 原则一：先定义状态，再定义动作

所有交互场景均需以“当前上下文状态”作为前提，而非孤立描述动作。

### 3.2 原则二：成员操作与容器操作分离

在所有场景中：

- “关闭文件夹”视为成员操作；
- “关闭工作区”视为容器操作。

### 3.3 原则三：交互结果应服从概念定义

交互层不得以简化界面为由，破坏以下概念结论：

1. 单文件夹是临时工作区；
2. 已保存工作区具有稳定身份；
3. 空工作区是合法状态。

---

## 4. 核心状态清单

交互场景涉及以下基础状态：

| 状态编号 | 状态名称 |
|---|---|
| S1 | 单文件夹临时工作区 |
| S2 | 多文件夹未保存工作区 |
| S3 | 已保存工作区（单文件夹或多文件夹） |
| S4 | 空工作区 |
| S5 | 无工作区 / 空白态 |

---

## 5. 核心交互场景

### 场景 1：打开单文件夹

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户以文件夹作为当前工作对象进入编辑上下文 |
| 前置条件 | 当前处于 S5 无工作区 / 空白态，或准备切换当前上下文 |
| 用户动作 | 通过“打开文件夹”选择一个根目录 |
| 系统响应 | 创建当前工作上下文，载入该根目录，展示该文件夹内容 |
| 结果状态 | 进入 S1 单文件夹临时工作区 |
| 状态栏要求 | 显示根文件夹名称 |
| 语义说明 | 此时用户虽感知为“打开了文件夹”，但产品抽象上已进入工作区上下文 |

### 场景 2：在单文件夹状态下保存工作区

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户将当前单文件夹上下文持久化为工作区 |
| 前置条件 | 当前处于 S1 单文件夹临时工作区 |
| 用户动作 | 执行“保存工作区”或“工作区另存为” |
| 系统响应 | 为当前上下文建立工作区文件身份 |
| 结果状态 | 进入 S3 已保存工作区（单文件夹） |
| 状态栏要求 | 由根文件夹名称切换为工作区名称 |
| 语义说明 | 从此开始，当前对象不再只是文件夹，而是具名工作区 |

### 场景 3：向当前上下文添加文件夹

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户扩展当前工作区的根文件夹集合 |
| 前置条件 | 当前处于 S1、S3 或 S4 |
| 用户动作 | 执行“将文件夹添加到工作区” |
| 系统响应 | 将新根目录加入当前工作区 |
| 结果状态 | S1 → S2 或 S3；S4 → S3 |
| 状态栏要求 | 若为已保存工作区，则持续显示工作区名称；若为未保存多文件夹，则显示“未命名工作区（未保存）” |
| 语义说明 | 该动作始终是向当前容器中增加成员，而非打开新模式 |

### 场景 4：单文件夹临时工作区中关闭文件夹

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户结束当前单文件夹上下文 |
| 前置条件 | 当前处于 S1 单文件夹临时工作区 |
| 用户动作 | 执行“关闭文件夹” |
| 系统响应 | 移除当前唯一根文件夹 |
| 结果状态 | 进入 S5 无工作区 / 空白态 |
| 状态栏要求 | 退出当前文件夹标识 |
| 语义说明 | 结果上可与“关闭工作区”一致，但其动作语义仍是成员操作 |

### 场景 5：单文件夹临时工作区中关闭工作区

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户结束当前工作上下文 |
| 前置条件 | 当前处于 S1 单文件夹临时工作区 |
| 用户动作 | 执行“关闭工作区” |
| 系统响应 | 结束当前工作区上下文 |
| 结果状态 | 进入 S5 无工作区 / 空白态 |
| 状态栏要求 | 退出当前文件夹标识 |
| 语义说明 | 结果上可与“关闭文件夹”一致，但动作语义是容器操作 |

### 场景 6：单文件夹已保存工作区中关闭文件夹

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户从已保存工作区中移除唯一根文件夹，但保留工作区本身 |
| 前置条件 | 当前处于 S3 已保存工作区，且仅包含一个根文件夹 |
| 用户动作 | 执行“关闭文件夹” |
| 系统响应 | 移除该唯一根文件夹，但不关闭工作区 |
| 结果状态 | 进入 S4 空工作区 |
| 状态栏要求 | 继续显示工作区名称 |
| 语义说明 | 该场景是“成员移除后容器保留”的关键场景，用于确认空工作区的合法性 |

### 场景 7：单文件夹已保存工作区中关闭工作区

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户结束整个已保存工作区上下文 |
| 前置条件 | 当前处于 S3 已保存工作区，且仅包含一个根文件夹 |
| 用户动作 | 执行“关闭工作区” |
| 系统响应 | 关闭当前工作区上下文 |
| 结果状态 | 进入 S5 无工作区 / 空白态 |
| 状态栏要求 | 退出当前工作区标识 |
| 语义说明 | 与场景 6 形成明确区分：此动作关闭容器，而非移除成员 |

### 场景 8：多文件夹工作区中关闭某个文件夹

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户从当前工作区中移除某个根文件夹 |
| 前置条件 | 当前处于 S2 或 S3，且包含多个根文件夹 |
| 用户动作 | 执行“关闭文件夹”并指定目标根文件夹 |
| 系统响应 | 移除目标根文件夹，保留其他根文件夹与当前工作区 |
| 结果状态 | 保持在 S2 或 S3；若移除后只剩一个根文件夹，则切换到单文件夹规则 |
| 状态栏要求 | 若为已保存工作区，显示工作区名称；若为未保存多文件夹，显示“未命名工作区（未保存）” |
| 语义说明 | 该动作不得隐式等同于关闭工作区 |

### 场景 9：多文件夹工作区中关闭工作区

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户结束整个多文件夹工作区上下文 |
| 前置条件 | 当前处于 S2 或 S3，且包含多个根文件夹 |
| 用户动作 | 执行“关闭工作区” |
| 系统响应 | 整体结束当前工作区上下文 |
| 结果状态 | 进入 S5 无工作区 / 空白态 |
| 状态栏要求 | 退出当前工作区标识 |
| 语义说明 | 与“关闭文件夹”必须保持稳定分离 |

### 场景 10：空工作区中重新添加文件夹

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户在保留工作区身份的前提下重新组织其成员 |
| 前置条件 | 当前处于 S4 空工作区 |
| 用户动作 | 执行“将文件夹添加到工作区” |
| 系统响应 | 向当前工作区加入新的根文件夹 |
| 结果状态 | 返回 S3 已保存工作区 |
| 状态栏要求 | 持续显示当前工作区名称 |
| 语义说明 | 该场景证明空工作区不是终止态，而是可继续操作的容器态 |

### 场景 11：空工作区中关闭工作区

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户结束空工作区上下文 |
| 前置条件 | 当前处于 S4 空工作区 |
| 用户动作 | 执行“关闭工作区” |
| 系统响应 | 关闭当前工作区上下文 |
| 结果状态 | 进入 S5 无工作区 / 空白态 |
| 状态栏要求 | 退出当前工作区标识 |
| 语义说明 | 空工作区虽不包含成员，但仍是完整的容器对象 |

### 场景 12：空工作区的页面空态表达

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户在空工作区中能够正确理解当前状态，并获得清晰下一步指引 |
| 前置条件 | 当前处于 S4 空工作区 |
| 用户动作 | 用户进入或停留在空工作区主界面 |
| 系统响应 | 在主区域展示空工作区专属空态内容 |
| 结果状态 | 保持在 S4 空工作区 |
| 状态栏要求 | 持续显示工作区名称 |
| 语义说明 | 该场景用于区分“空工作区”与“无工作区 / 空白态” |

该场景下，页面空态需满足以下定义：

#### 标题

推荐标题：

> **工作区当前未包含文件夹**

#### 说明文案

推荐说明：

> **你仍位于当前工作区中。可继续将文件夹添加到该工作区，或关闭当前工作区。**

#### 动作层级

- 主动作：**将文件夹添加到工作区**
- 次动作：**关闭工作区**

补充约束：

1. `打开文件夹`可保留在菜单路径中，但不作为空态主区域的主动作；
2. 页面表达不得将空工作区描述为错误、失败或异常；
3. 页面表达应强调“当前工作区仍存在，只是暂未包含文件夹”。

### 场景 13：空工作区下的菜单可用性

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户在空工作区中仍可通过菜单继续组织当前工作区 |
| 前置条件 | 当前处于 S4 空工作区 |
| 用户动作 | 打开 File 菜单 |
| 系统响应 | 展示与空工作区一致的菜单动作集合 |
| 结果状态 | 保持在 S4 空工作区，直到用户执行后续动作 |
| 状态栏要求 | 持续显示工作区名称 |
| 语义说明 | 菜单动作必须服从“工作区仍存在，但成员为空”的对象关系 |

该场景下，File 菜单应保持固定结构，并满足以下可用性要求：

- 打开文件…
- 打开文件夹…
- 打开工作区…
- 打开最近
- 将文件夹添加到工作区…
- 保存工作区
- 工作区另存为…
- 关闭工作区

该场景下，以下动作应处于**不可用**状态：

- 关闭文件夹

原因如下：

1. 当前不存在可操作的根文件夹成员；
2. “关闭文件夹”属于成员操作，不应在无成员时保留为可执行动作；
3. 该约束有助于稳定区分成员操作与容器操作；
4. 菜单结构稳定性要求优先通过“可用 / 不可用”表达状态差异，而非通过显隐表达。

### 场景 14：File 菜单在不同状态下的稳定性

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户在不同工作区状态下打开 File 菜单时，保持对菜单结构的稳定认知 |
| 前置条件 | 当前处于 S1、S2、S3、S4 或 S5 任一状态 |
| 用户动作 | 打开 File 菜单 |
| 系统响应 | 以固定结构展示菜单项，并根据状态控制动作可用性 |
| 结果状态 | 当前状态保持不变，直到用户执行具体动作 |
| 状态栏要求 | 与当前状态保持一致 |
| 语义说明 | 菜单作为能力地图应保持稳定，状态差异优先通过可用/不可用表达 |

该场景下，菜单需满足以下交互要求：

#### 1. 结构稳定

File 菜单中的核心动作应保持固定顺序与固定命名，不因当前状态频繁改名或重排。

#### 2. 状态差异表达方式

当前状态差异应优先通过“可用 / 不可用”表达，而不是通过菜单项显隐表达。

#### 3. 关键动作可用性规则

| 动作 | S1 单文件夹临时工作区 | S2 多文件夹未保存工作区 | S3 已保存工作区 | S4 空工作区 | S5 无工作区 / 空白态 |
|---|---|---|---|---|---|
| 将文件夹添加到工作区 | 可用 | 可用 | 可用 | 可用 | 不可用 |
| 保存工作区 | 可用 | 可用 | 可用 | 可用 | 不可用 |
| 工作区另存为 | 可用 | 可用 | 可用 | 可用 | 不可用 |
| 关闭文件夹 | 可用 | 可用 | 可用 | 不可用 | 不可用 |
| 关闭工作区 | 可用 | 可用 | 可用 | 可用 | 不可用 |

#### 4. 命名稳定性要求

当前阶段不对以下动作做状态化改名：

- 保存工作区
- 工作区另存为
- 关闭文件夹
- 关闭工作区

该场景的核心原则为：

> **菜单命名稳定，菜单结构稳定，状态差异通过动作是否可用来表达。**

### 场景 15：打开最近菜单的分组展示

| 项目 | 内容 |
|---|---|
| 场景目标 | 用户在“打开最近”中快速识别不同类型的最近记录 |
| 前置条件 | 系统中已存在最近打开历史 |
| 用户动作 | 打开 `文件 > 打开最近` |
| 系统响应 | 以稳定的三组结构展示最近记录 |
| 结果状态 | 当前状态保持不变，直到用户点击某条记录 |
| 状态栏要求 | 与当前上下文保持一致 |
| 语义说明 | 最近打开用于按对象类型组织历史，而不是按底层来源重复罗列 |

该场景下，菜单分组结构应为：

1. **工作区**
2. **文件夹**
3. **文件**

各组含义如下：

| 分组 | 含义 |
|---|---|
| 工作区组 | 具名、可持久化的工作区对象历史 |
| 文件夹组 | 文件夹入口历史，包括单文件夹临时工作区 |
| 文件组 | 最近打开的文件历史 |

### 场景 16：最近打开中的唯一归属规则

| 项目 | 内容 |
|---|---|
| 场景目标 | 保证最近打开中的对象归属清晰、不重复 |
| 前置条件 | 最近打开记录中同时存在工作区、文件夹和文件对象 |
| 用户动作 | 浏览“打开最近”菜单 |
| 系统响应 | 每个对象仅出现在一个最合适的分组中 |
| 结果状态 | 当前状态保持不变 |
| 状态栏要求 | 与当前上下文保持一致 |
| 语义说明 | 对象归属应服务于理解，不应为了兼容旧习惯而重复展示 |

该场景下，归属规则定义如下：

| 对象类型 | 所属分组 |
|---|---|
| 已保存工作区 | 工作区组 |
| 空工作区对应的已保存工作区 | 工作区组 |
| 单文件夹临时工作区 | 文件夹组 |
| 普通最近打开文件夹 | 文件夹组 |
| 最近打开文件 | 文件组 |

该场景下必须满足：

1. 单文件夹临时工作区不得同时出现在工作区组和文件夹组；
2. 不再采用历史版本中的双重归属策略；
3. 同一对象在同一菜单中不得重复出现；
4. 当存在重名项时，应提供补充信息帮助区分，但不改变其分组归属。

## 7. 交互场景结论

当前阶段围绕工作区的关键交互路径可归纳为：

1. 单文件夹不是工作区之外的例外，而是工作区状态之一；
2. 已保存工作区在单文件夹场景下也具有独立容器身份；
3. 空工作区是“关闭文件夹但不关闭工作区”的必要结果状态；
4. 所有交互场景都必须稳定地区分成员操作与容器操作；
5. 交互设计必须服从概念与需求，而不得反向修改概念边界。
