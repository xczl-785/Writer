## 2. 文档目标

本文档用于将已确认的需求与语义边界转化为可直接执行的实施计划，并明确“先溜边、后完整实施”的推进方式。

本文档不承担新增需求定义职责，而是将以下内容系统化：

1. 当前阶段的**溜边规则方向**；
2. 当前实现与目标语义之间的**断层清单**；
3. 后续编码与测试的**模块实施顺序**；
4. 执行期应遵守的**验收与回归边界**。

---

## 3. 总体实施策略

当前阶段采用两层推进：

1. **先做溜边收口**：先将状态、对象层级、菜单可用性矩阵、关闭语义、Open Recent 分组规则、主区空态表达与当前代码断层统一为一套实现约束；
2. **再做模块化实施**：按“状态模型 → 菜单可用性 → 命令行为 → 主区/状态栏 → Open Recent → 测试”顺序推进，避免边做边补规则。

本轮的固定设计前提如下：

- 采用**隐式工作区**：底层统一为工作区模型，表层仍以文件夹、文件操作为主；
- **S4 空工作区** 与 **S5 无工作区** 必须严格区分；
- `关闭文件夹` 仅对**当前明确选中的根文件夹**生效；
- `关闭文件` 纳入本轮，且目标固定为**当前活动文件**；
- `打开文件夹` 与 `将文件夹添加到工作区` 同时保留，但分别承担“进入/替换上下文”与“修改当前工作区成员”的不同语义。

## 4. 溜边规则方向

### 4.1 状态边界方向

后续所有实现与测试，统一使用以下五种状态：

| 状态 | 定义 |
|---|---|
| S1 | 单文件夹临时工作区 |
| S2 | 多文件夹未保存工作区 |
| S3 | 已保存工作区 |
| S4 | 空工作区（已保存、0 根文件夹） |
| S5 | 无工作区 / 空白态 |

溜边要求：

1. 不再允许仅用 `folders.length > 0` 近似表达“是否在工作区”；
2. `workspaceFile != null && folders.length === 0` 必须明确识别为 **S4**；
3. `workspaceFile == null && folders.length === 0` 必须明确识别为 **S5**；
4. 单根已保存工作区与单根临时工作区必须区分，不得共用命名与关闭语义。

### 4.2 对象层级方向

File 菜单与相关交互统一按以下三层对象语义组织：

| 层级 | 动作 |
|---|---|
| 文件级 | 打开文件、关闭文件 |
| 成员级 | 打开文件夹、关闭文件夹、将文件夹添加到工作区 |
| 容器级 | 打开工作区、保存工作区、工作区另存为、关闭工作区 |

溜边要求：

1. 禁止将成员级动作与容器级动作混用；
2. 禁止将 `关闭文件夹` 实现为关闭整个工作区；
3. 禁止将 `关闭工作区` 实现为移除某个根文件夹；
4. 文件级、成员级、容器级动作在命名和禁用态上必须有稳定边界。

### 4.3 File 菜单可用性方向

后续实现统一按以下矩阵：

#### S1 / S2 / S3
可用：
- 打开文件
- 打开文件夹
- 打开工作区
- 打开最近
- 关闭文件（有活动文件时）
- 关闭文件夹（满足根节点选择前提时）
- 保存工作区
- 工作区另存为
- 关闭工作区
- 将文件夹添加到工作区

#### S4
可用：
- 打开文件
- 打开文件夹
- 打开工作区
- 打开最近
- 保存工作区
- 工作区另存为
- 关闭工作区
- 将文件夹添加到工作区

禁用：
- 关闭文件夹
- 关闭文件（无活动文件时）

#### S5
可用：
- 打开文件
- 打开文件夹
- 打开工作区
- 打开最近

禁用：
- 保存工作区
- 工作区另存为
- 将文件夹添加到工作区
- 关闭文件夹
- 关闭工作区
- 关闭文件

补充规则：

1. `关闭文件` 仅在存在活动文件时可用；
2. `关闭文件夹` 仅在 `selectedPath` 恰好等于某个根文件夹路径时可用；
3. 不允许对子目录、文件进行“自动上推”来启用 `关闭文件夹`；
4. 状态差异优先通过可用 / 不可用表达，不通过频繁隐藏菜单项表达。

### 4.4 主区空态方向

后续主区表达必须至少区分两类：

#### S4：空工作区
- 当前仍处于某个工作区中；
- 状态栏继续显示工作区名；
- 主动作：`将文件夹添加到工作区`；
- 次动作：`关闭工作区`；
- 不应退化成普通欢迎页。

#### S5：无工作区 / 空白态
- 当前不存在工作区上下文；
- 状态栏不显示工作区名；
- 主区使用欢迎页 / 空白页表达；
- 主动作维持 `打开文件夹`、`打开工作区` 的入口逻辑。

### 4.5 Open Recent 方向

`打开最近` 统一采用稳定三组结构：

1. 工作区
2. 文件夹
3. 文件

溜边要求：

1. 分组顺序固定；
2. 分组语义固定；
3. 同一对象只能唯一归属一个分组；
4. 空组允许隐藏，但不得改变整体分组含义。

## 5. 当前实现断层清单

### 5.1 状态模型断层

当前代码中：

- `workspaceStore.getWorkspaceType()` 仅返回 `empty | single | multi`；
- 无法表达 **S4 空工作区** 与 **S5 无工作区** 的区别；
- 单根已保存工作区与单根临时工作区也缺少清晰分层。

### 5.2 状态栏断层

当前代码中：

- 单根时优先显示文件夹名；
- `folders.length === 0` 时直接不显示工作区标识；
- 无法满足：
  - 单根已保存工作区显示工作区名；
  - 空工作区显示工作区名。

### 5.3 File 菜单断层

当前代码中：

- `打开最近` 位于 Tools 而非 File；
- File 菜单未形成“文件 / 工作区 / 导出 / 关闭”的稳定结构；
- `关闭文件` 尚未形成明确落点；
- 原生菜单与自绘菜单的结构尚未同步到本轮方案。

### 5.4 命令行为断层

当前代码中：

- `menu.file.close_folder` 直接执行 `workspaceActions.closeWorkspace()`；
- 这与“成员级动作”语义冲突；
- `save_workspace` / `save_workspace_as` 基于 `folders.length > 0` 判断，无法支持 S4。

### 5.5 主区表达断层

当前代码中：

- 空态主要依据“有无文件夹”粗略判断；
- 尚未把 S4 与 S5 区分为两类空态。

## 6. 模块实施计划

### 阶段 A：状态模型与派生能力

目标：先统一底层状态判断，为后续所有菜单、状态栏、主区和命令逻辑提供同一事实来源。

实施内容：

1. 在工作区状态层新增可推导能力，用于判断：
   - 是否存在工作区上下文；
   - 是否为已保存工作区；
   - 是否为空工作区；
   - 是否为单根临时工作区；
   - 是否为多根未保存工作区；
2. 继续保留 `workspaceFile` 作为“已保存工作区身份”的主依据；
3. 保留 `selectedPath` 作为 `关闭文件夹` 的执行前提来源；
4. 禁止在组件层散落重复状态判断，后续一律依赖派生 helper。

阶段完成标准：

- 可以稳定判断 S1~S5；
- 可以稳定判断“当前是否具备工作区上下文”；
- 可以稳定判断“当前是否具备关闭文件夹资格”。

### 阶段 B：File 菜单结构与可用性重建

目标：将 File 菜单从历史结构迁移为本轮确认后的稳定结构，并让菜单 enable/disable 逻辑与状态模型对齐。

实施内容：

1. 重建 `menuSchema` 中的 File 菜单结构；
2. 将 `打开最近` 从 Tools 迁回 File；
3. 为 File 菜单补齐：
   - 打开文件
   - 打开工作区
   - 将文件夹添加到工作区
   - 关闭文件
   - 关闭工作区
4. 重写 `menuState`，改为基于：
   - 工作区状态枚举
   - 当前活动文件存在性
   - 当前根节点选择态
   共同推导；
5. 同步更新原生菜单，使其与自绘菜单项目、顺序、默认禁用态一致。

阶段完成标准：

- File 菜单结构稳定；
- 同一动作在原生菜单与自绘菜单中定义一致；
- S1~S5 下菜单可用性符合确认矩阵。

### 阶段 C：命令层与行为语义修正

目标：将菜单点击行为与最终语义完全对齐。

实施内容：

1. `关闭文件`
   - 关闭当前活动文件；
   - 无活动文件时不执行；
   - 关闭最后一个文件后仅保留当前工作区上下文，不自动关闭工作区；
2. `关闭文件夹`
   - 仅对当前选中的根文件夹执行；
   - S1：结果进入 S5；
   - S3 单根：结果进入 S4；
   - 多根：仅移除该成员并保留工作区；
3. `关闭工作区`
   - 始终关闭整个工作区容器；
   - S1 / S2 / S3 / S4 均进入 S5；
4. `保存工作区` / `工作区另存为`
   - 允许在 S4 下执行；
   - S5 禁用；
5. `打开文件夹`
   - 明确为进入或替换当前上下文；
6. `将文件夹添加到工作区`
   - 明确为修改当前工作区成员；
   - S4 可用，S5 禁用。

阶段完成标准：

- 菜单命令行为与确认项文档完全一致；
- 不再存在“成员级动作调用容器级关闭”的逻辑错误；
- S4 的保存与关闭路径可以稳定执行。

### 阶段 D：状态栏、主区空态与用户可见表达修正

目标：保证用户可见表达与隐式工作区设计一致，即“底层统一，表层自然”。

实施内容：

1. 状态栏命名规则统一为：
   - S1：文件夹名；
   - 单根已保存工作区 / S3 / S4：工作区名；
   - S2：未命名工作区（未保存）；
   - S5：不显示工作区标识；
2. 主区空态拆分为：
   - S4 专属空工作区页面；
   - S5 欢迎页 / 空白态页面；
3. S4 页面中：
   - 主动作：将文件夹添加到工作区；
   - 次动作：关闭工作区；
4. 保持“打开文件夹”“打开工作区”仍作为菜单入口，而不是 S4 主区主按钮。

阶段完成标准：

- 用户能清晰区分“当前没有工作区”与“当前工作区为空”；
- 状态栏命名与工作区身份保持一致；
- S4 不再被误表达为欢迎页。

### 阶段 E：Open Recent 与集成收尾

目标：完成 File 菜单路径、Open Recent 展示、主区切换和回归测试的整合。

实施内容：

1. 沿用现有 `RecentItemsService` 存储结构，不新增持久化模型；
2. 调整菜单承接路径，让 `打开最近` 以 File 菜单入口呈现；
3. 保持 Recent 菜单中的三组稳定结构；
4. 补齐 App 层与菜单层的集成行为验证。

阶段完成标准：

- `打开最近` 可从 File 菜单稳定触达；
- 最近项分组与唯一归属规则不被破坏；
- 主区、状态栏、菜单路径之间不再出现互相矛盾的状态表达。

## 7. 推荐执行顺序

建议按以下顺序执行，不建议打乱：

1. **状态模型与 helper**
2. **菜单可用性推导**
3. **命令层行为修正**
4. **状态栏与主区空态**
5. **Open Recent 路径与集成**
6. **测试补全与回归清理**

排序理由：

- 状态模型不稳定时，菜单和 UI 只能写临时判断；
- 菜单可用性不稳定时，命令行为很容易出现入口与逻辑不一致；
- 命令行为未收口前，不应先做大量 UI 细节打磨；
- Open Recent 属于最后的整合项，适合放在主逻辑完成后收尾。

## 9. 执行边界与默认决策

### 9.1 本轮执行边界

1. 不新增额外收口文档；
2. 不重写工作区文件格式；
3. 不扩展导出能力本身，仅整理其在 File 菜单中的归位；
4. 不将“隐式工作区”转为用户强显式概念；
5. 不以临时兼容逻辑换取短期通过，而应优先统一派生层与命令层。

### 9.2 默认决策

1. 后续实现以本计划文档与确认项文档作为共同依据；
2. 后续编码期优先修改派生层与命令层，不在组件层散落重复分支；
3. 状态差异优先通过矩阵化规则落地，而不是通过零散 if/else 临时拼接；
4. 若实施过程中发现确认项冲突，应暂停实现并回到确认项文档追加修订记录。

## 10. 计划结论

当前阶段最合理的推进路径是：

1. 先完成**状态与对象语义的溜边收口**；
2. 再按模块顺序推进实现；
3. 最后统一补齐测试与原生 / 自绘菜单一致性回归。

该路径的核心收益在于：

- 避免在历史逻辑上继续叠加临时分支；
- 避免边做边补需求口径；
- 让“隐式工作区、底层统一、表层自然”的方向在实现上真正落地。
