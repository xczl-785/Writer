## 2. 文档摘要

本文档用于统一 Writer 当前阶段关于“单文件夹、多文件夹、工作区”的产品概念、状态分层、名称显示规则与关闭语义边界。

本文档仅对**产品语义与文档结构**进行定义，不涉及以下内容：

- 不修改既有 `.writer-workspace` 文件概念；
- 不修改既有 `.writer-workspace` 文件格式；
- 不扩展新的功能能力；
- 不进入具体实现方案与代码结构设计；
- 不讨论代码实现成本、实现路径与工程取舍。

本文档的核心目标如下：

1. 统一“文件夹”与“工作区”的抽象边界；
2. 明确单文件夹与多文件夹场景下的状态定义；
3. 明确状态栏显示名称规则；
4. 明确“关闭文件夹”与“关闭工作区”的产品语义差异。

---

## 3. 背景与问题陈述

当前 Writer 在文件、文件夹、工作区三个对象层级上，整体交互方向已与 VS Code 接近，但仍存在阶段性过渡痕迹。当前需要重点对齐的问题包括：

1. 打开单文件夹时，当前上下文是否属于工作区；
2. 单文件夹与多文件夹场景下，底部状态栏名称应如何显示；
3. “保存工作区”“关闭文件夹”“关闭工作区”等操作在不同场景中的语义边界应如何理解；
4. 在不修改既有 `.writer-workspace` 文件概念与格式的前提下，如何统一产品表达。

因此，本文档的定位不是重构工作区模型，而是：

> 在保持现有工作区文件定义不变的前提下，明确当前阶段的产品语义，为菜单整理、状态栏展示、关闭行为与后续需求分析提供统一依据。

## 4. 约束与前提

### 4.1 现有工作区文件基础

Writer 当前已存在工作区保存文件，扩展名为 `.writer-workspace`。其核心结构已包含：

- `version`
- `folders[]`
- `state.openFiles`
- `state.activeFile`
- `state.sidebarVisible`

这表明产品当前已经具备“工作区是一个可持久化上下文容器”的基础能力。

### 4.2 当前阶段约束

本阶段**不建议修改工作区定义本体，也不建议修改既有文件格式**。原因如下：

1. 已有保存工作区的概念和文件格式已存在，修改成本高；
2. 当前问题主要集中在产品语义、命名与交互一致性，而非数据模型缺失；
3. 通过补充状态分层和显示规则，已可解决当前讨论中的主要歧义。

当前阶段采取的总体策略如下：

> **保持现有工作区模型不变，补充单文件夹、多文件夹、已保存工作区之间的产品语义定义。**

## 5. 核心概念定义

### 5.1 工作区（Workspace）

工作区是 Writer 中用于承载当前工作上下文的统一容器。其本质不是某一个文件，也不是某一个文件夹，而是“当前这组工作对象及其状态的组织结果”。

工作区至少包含以下两类内容：

1. **根文件夹集合**：一个或多个根目录；
2. **工作状态**：例如当前打开文件、激活文件、侧边栏状态等。

据此，工作区是一个**抽象上的统一概念**，而不是仅在“打开 `.writer-workspace` 文件”时才成立。

### 5.2 单文件夹（Single-folder Context）

当用户通过“打开文件夹”进入某个目录时，系统进入的是：

> **单文件夹临时工作区**

该状态具备以下特征：

1. 当前上下文在产品抽象上仍属于工作区；
2. 该工作区仅包含一个根文件夹；
3. 该工作区当前尚未被显式保存为 `.writer-workspace` 文件；
4. 用户心智更倾向于“我打开了一个项目文件夹”，而非“我打开了一个工作区文件”。

因此，单文件夹场景既不应被视为“非工作区”，也不应被误判为“已保存工作区”。其准确表述应为：

> **单文件夹 = 临时工作区的一种特殊形态**

### 5.3 多文件夹（Multi-root Context）

当当前上下文中包含两个及以上根文件夹时，系统进入的是：

> **多根工作区**

该状态具备以下特征：

1. 工作区不再可自然等同于某一个根文件夹；
2. 用户感知对象从“某个文件夹”转向“这一组被组织起来的内容”；
3. 若尚未保存，则该工作区应被视为“未命名工作区”；
4. 若已保存，则应以工作区名称作为主要标识。

多文件夹场景下，“工作区”概念对用户而言已是显性概念，而不再只是系统内部抽象。

### 5.4 已保存工作区（Saved Workspace）

当当前工作上下文被用户显式保存为 `.writer-workspace` 文件后，进入：

> **已保存工作区**

其关键特征如下：

1. 具有明确的工作区文件路径；
2. 具有可复用、可再次打开的稳定身份；
3. 对外显示名称优先取工作区名称，而不是某一个根文件夹名称；
4. 无论其内部包含一个还是多个根文件夹，均属于“已保存工作区”。

这意味着：

> **“是否保存为工作区文件”与“包含几个根文件夹”是两个不同维度。**

### 5.5 空工作区（Empty Saved Workspace）

当一个已保存工作区中的根文件夹被全部移除，但该工作区文件本身仍作为当前上下文继续存在时，进入：

> **空工作区**

其关键特征如下：

1. 该状态仅适用于**已保存工作区**；
2. 当前工作区不包含任何根文件夹；
3. 当前工作区名称、身份与工作区文件路径仍然有效；
4. 用户仍处于该工作区上下文中，可继续向其中添加文件夹；
5. 该状态用于体现“工作区是容器，文件夹是成员”这一产品语义。

## 6. 状态模型定义

### 6.1 状态分层

为避免“文件夹模式”和“工作区模式”并存造成的心智分裂，当前阶段建议统一为一个连续状态模型。

Writer 当前阶段可将上下文划分为以下四类产品状态：

| 状态 | 定义 | 根文件夹数量 | 是否有工作区文件 | 用户主要感知 |
|---|---|---:|---|---|
| 单文件夹临时工作区 | 通过“打开文件夹”进入的未保存上下文 | 1 | 否 | 一个文件夹 / 一个项目 |
| 多文件夹未保存工作区 | 包含多个根文件夹但尚未保存的上下文 | ≥2 | 否 | 一组内容组成的工作区 |
| 空工作区 | 已保存工作区暂时不包含任何根文件夹 | 0 | 是 | 一个仍然存在的工作区容器 |
| 已保存工作区 | 已保存为 `.writer-workspace` 的上下文 | 1~N | 是 | 一个具名工作区 |

### 6.2 状态迁移

三者并非相互割裂，而是连续演进关系：

1. 打开文件夹 → 进入单文件夹临时工作区；
2. 在当前上下文中添加其他文件夹 → 演进为多文件夹未保存工作区；
3. 执行“保存工作区”或“工作区另存为” → 演进为已保存工作区；
4. 在已保存工作区中移除全部根文件夹 → 演进为空工作区；
5. 在空工作区中添加文件夹 → 返回已保存工作区。

当前阶段最重要的统一表达如下：

> **用户始终处于某个工作区上下文中，只是该工作区的保存状态和根数量不同。**

## 7. 显示规则定义

### 7.1 状态栏命名职责

底部状态栏不应机械反映内部数据类型，而应优先承担“当前工作上下文标识”的职责。

### 7.2 推荐显示规则

| 场景 | 状态栏名称显示 |
|---|---|
| 单文件夹临时工作区 | 根文件夹名称 |
| 单文件夹且已保存为工作区 | 工作区名称 |
| 空工作区 | 工作区名称 |
| 多文件夹未保存工作区 | 未命名工作区（未保存） |
| 多文件夹已保存工作区 | 工作区名称 |

### 7.3 规则解释

单文件夹时，用户最自然的识别对象是“这个文件夹 / 这个项目”，因此显示根文件夹名称是合理的。

多文件夹时，当前上下文无法再由某一个根目录代表，因此应切换为工作区命名语义。

当工作区被保存后，用户识别的主要对象应变为“工作区文件本身”，此时名称展示应优先采用工作区名称，而不是某个根目录名称。

当工作区处于空工作区状态时，虽然不再包含根文件夹，但工作区容器本身仍然存在，因此状态栏仍应显示工作区名称，而不应退化为空白或无上下文命名。

因此，状态栏命名规则可归纳为：

> **优先显示用户对当前上下文最容易形成稳定识别的名称。**

该规则允许单文件夹显示文件夹名，同时不否认其内部仍属于工作区抽象。

## 8. 操作语义定义

### 8.1 打开文件夹

“打开文件夹”不是进入一种与工作区无关的特殊模式，而是创建一个只含单根目录的临时工作区上下文。

其产品语义包括：

- 创建当前工作上下文；
- 载入一个根文件夹；
- 尚未形成持久化工作区文件；
- 可进一步扩展为多文件夹工作区；
- 可进一步保存为工作区。

### 8.2 打开工作区

“打开工作区”是指打开一个已存在的 `.writer-workspace` 文件，并恢复其已保存的上下文定义与工作状态。

其产品语义包括：

- 打开具名、可复用的工作区；
- 恢复其包含的根文件夹集合；
- 恢复与工作区相关的已保存状态。

### 8.3 将文件夹添加到工作区

该操作的语义应理解为：

> **在当前工作区上下文中新增一个根文件夹**

场景说明如下：

- 当当前为单文件夹临时工作区时，此操作会使其演进为多文件夹未保存工作区；
- 当当前为已保存工作区时，此操作会改变该工作区内容，并使其进入已修改但未重新保存的状态。

### 8.4 保存工作区

当前阶段建议保留“保存工作区”能力，且**单文件夹临时工作区应支持保存工作区**。

理由如下：

1. 单文件夹本质上已属于临时工作区；
2. 若单文件夹不允许保存工作区，则“打开文件夹 → 持久化当前上下文”这条路径会被人为切断；
3. 从用户心智上，用户可能希望把当前项目上下文保存为一个可再次打开的工作区，即使该工作区只包含一个文件夹；
4. 从产品模型上，这能保持从临时态到持久态的统一演进关系。

因此，当前阶段建议：

> **只要存在文件夹上下文，就允许执行“保存工作区”。**

### 8.5 关闭文件夹与关闭工作区

“关闭文件夹”与“关闭工作区”在当前阶段应被定义为两个不同层级的动作：

- **关闭文件夹**：面向根文件夹对象；
- **关闭工作区**：面向工作区上下文容器。

两者在单文件夹场景下可能产生相同结果，但在产品语义上不应被定义为同一个动作；在多文件夹场景下，两者必须严格区分。

#### 8.5.1 单文件夹场景

当当前上下文为单文件夹临时工作区时：

- **关闭文件夹**：移除当前唯一根文件夹；
- **关闭工作区**：关闭当前工作区上下文。

由于当前工作区仅包含一个根文件夹，因此二者在当前阶段的**执行结果可以一致**，即：

> **关闭后均返回无文件夹 / 空白态**

但从产品语义上仍应保留区分：

1. “关闭文件夹”操作对象是当前根文件夹；
2. “关闭工作区”操作对象是当前上下文容器；
3. 当前仅因单文件夹场景下二者边界重合，才表现为结果一致。

据此，当前阶段建议：

> **单文件夹场景下，概念上区分语义，执行上允许复用同一关闭结果。**

#### 8.5.2 单文件夹已保存为工作区场景

当当前上下文为“单文件夹且已保存为工作区”时，不再适合将“关闭文件夹”与“关闭工作区”定义为结果一致的动作。

原因如下：

1. 此时工作区已经具备独立稳定身份；
2. 用户当前操作对象不再只是某个文件夹，而是一个具名、可复用的工作区；
3. 若仍将“关闭文件夹”直接等同于“关闭工作区”，会破坏“容器”与“成员”的产品边界。

因此，当前阶段推荐采用以下定义：

**关闭文件夹**

> 从当前已保存工作区中移除唯一根文件夹

其结果为：

1. 当前根文件夹被移除；
2. 工作区本身不关闭；
3. 当前上下文转为空工作区；
4. 状态栏继续显示工作区名称；
5. 用户后续仍可向该工作区重新添加文件夹。

**关闭工作区**

> 关闭整个已保存工作区上下文

其结果为：

1. 当前工作区整体关闭；
2. 系统退出该工作区上下文；
3. 返回无工作区 / 空白态。

据此，当前阶段结论为：

> **单文件夹已保存为工作区后，“关闭文件夹”与“关闭工作区”应同时在语义与结果层分离。**

#### 8.5.3 多文件夹场景

当当前上下文为多文件夹工作区时，两者必须严格区分。

**关闭文件夹** 的语义定义为：

> **从当前工作区中移除某一个根文件夹**

执行结果如下：

1. 被关闭的根文件夹从当前工作区中移除；
2. 其他根文件夹继续保留；
3. 当前工作区继续存在；
4. 若该工作区已保存，则工作区进入“已修改未保存”状态。

**关闭工作区** 的语义定义为：

> **关闭整个工作区上下文**

执行结果如下：

1. 当前工作区整体关闭；
2. 其中包含的所有根文件夹一并退出；
3. 系统返回无工作区 / 空白态。

因此，在多文件夹场景下：

> **关闭文件夹是成员级操作，关闭工作区是容器级操作，两者不得混用。**

#### 8.5.4 最后一个根文件夹的处理原则

在多文件夹工作区中，若通过连续关闭文件夹操作使根文件夹数量逐步减少，则需要明确“最后一个根文件夹”的处理边界。

当前阶段推荐采用以下原则：

> **当上下文仅剩最后一个根文件夹时，不再把该动作按多文件夹场景下的普通“关闭文件夹”语义处理，而是切换到单文件夹场景规则。**

采用该原则的原因如下：

1. 避免“关闭文件夹”在不加区分的情况下悄然演化为“关闭工作区”；
2. 保持“成员操作”与“容器操作”的概念边界稳定；
3. 为菜单动态展示、后续测试与实现提供更清晰的判断条件。

当前阶段推荐方案为：

- 多文件夹场景下，同时提供“关闭文件夹”与“关闭工作区”；
- 当上下文仅剩最后一个根文件夹时，进入单文件夹场景语义；
- 不建议将“关闭文件夹”无差别执行到根文件夹数量为 0 的状态，再隐式等同于“关闭工作区”。

## 9. 当前阶段结论

### 9.1 已确认结论

1. 工作区是统一抽象，不应仅在打开 `.writer-workspace` 文件时成立；
2. 单文件夹属于临时工作区，而非“非工作区”；
3. 多文件夹是工作区的显性形态；
4. 已保存工作区是工作区的持久化形态；
5. 状态栏名称应按“当前最易识别的上下文名称”显示；
6. 单文件夹场景应允许“保存工作区”；
7. 单文件夹场景下，“关闭文件夹”与“关闭工作区”在结果上可一致，但语义上仍应区分；
8. 单文件夹已保存为工作区后，“关闭文件夹”与“关闭工作区”应在结果与语义上同时分离；
9. 空工作区是已保存工作区的一种合法状态；
10. 多文件夹场景下，“关闭文件夹”与“关闭工作区”必须严格区分；
11. 当前阶段不修改 `.writer-workspace` 定义与格式。

## 11. 总结

Writer 当前阶段关于单文件夹、多文件夹与工作区的最优定义，不是拆成三套互斥模式，而是将其统一纳入“工作区上下文”的连续状态模型。

其核心总结如下：

- 单文件夹是**临时工作区**；
- 多文件夹是**未保存或已保存的多根工作区**；
- 空工作区是**已保存工作区的合法容器状态**；
- `.writer-workspace` 是工作区的**持久化载体**；
- 状态栏名称显示遵循“用户最容易识别当前上下文”的原则；
- “关闭文件夹”与“关闭工作区”需要在概念层保持稳定分层。

该定义能够在不修改当前工作区文件格式的前提下，最大程度统一产品心智、菜单语义与后续交互设计。
