# AI 辅助迭代开发一致性问题复盘与落地方案

**日期**：2026-03-19  
**适用项目**：Writer  
**文档目的**：用于梳理当前 AI 辅助迭代开发中出现的一致性问题，明确根本原因，并提出轻量、可持续执行的落地方案，作为后续继续讨论与执行的基线。

---

## 1. 背景与问题定义

Writer 项目从灵感、产品定义、技术选型到迭代开发，均大量依赖 AI agent 参与完成。随着版本持续推进，项目出现了一个典型现象：**同一能力在不同入口、不同版本、不同补丁中逐步分叉，最终表现为行为不一致、边界失配、局部修复难以收束。**

本次“新建文件 / 新建文件夹”能力的不统一问题即是代表性案例。该问题并非单点失误，而是 AI 驱动持续迭代过程中，工程知识组织方式与工作流约束不足的集中体现。

因此，本次复盘的关注点不是某一个功能是否写错，而是：

1. 为什么 AI 在版本迭代中越来越难把握项目细节；
2. 为什么结构文档（如 UML）不足以防止行为分叉；
3. 如何为冷启动 agent 提供更高密度、更可执行的项目上下文；
4. 如何把这些上下文真正转化为迭代开发中的执行约束。

---

## 2. 当前现象的根本原因

### 2.1 第一层：知识类型错配

当前项目中，AI 更容易直接看到的是：

- 代码实现；
- 模块结构；
- 静态依赖关系；
- 局部函数与组件之间的调用链。

但实际导致行为不一致的问题，往往并不发生在“结构层”，而发生在以下知识层：

- 能力边界；
- 行为定义；
- 入口一致性约束；
- 历史决策；
- 影响范围；
- 产品语义与工程实现之间的对应关系。

换言之，项目当前沉淀较多的是**结构知识**，而缺少足够密集、足够明确的**行为知识与决策知识**。  
这导致 AI 能理解“代码是怎么组织的”，却无法高效理解“为什么必须这样做、哪些地方必须同步变化、哪些差异本身就是错误”。

### 2.2 第二层：知识载体错配

目前项目中的关键上下文，实际上分散在多个载体中：

- 人与 AI 的对话；
- 临时分析报告；
- 某次 bug 修复上下文；
- 局部代码实现；
- 人的记忆与判断。

这些信息在单次任务中是可见的，但对下一次冷启动 agent 来说并不稳定。  
也就是说，项目中的关键信息并没有集中沉淀为一种**可长期读取、可重复消费、可用于约束后续开发**的工程知识载体。

结果就是：

- 当前 agent 了解细节；
- 下一个 agent 只能重新猜测；
- 多个版本之后，历史共识逐步漂移。

### 2.3 第三层：工作流错配

即便存在一些说明性文档，如果工作流没有强制要求 AI 在实现前完成以下动作：

1. 查询相关能力；
2. 查询已有约束；
3. 判断是否影响其他入口；
4. 在实现后回写关键文档；

那么 AI 仍会回到默认模式：

- 读取局部代码；
- 解决当前任务；
- 在最近、最显眼的入口上补逻辑；
- 缺少对“同一能力其他入口是否同步”的系统性检查。

因此，问题并不只是“缺文档”，而是**缺少把文档转化为执行前置条件的工作流约束**。

## 3. 对 UML 与单一事实源方案的评价

### 3.1 UML 的作用边界

`docs/generated/` 下的 UML / PlantUML 自动图，主要解决的是以下问题：

- 帮助冷启动 agent 快速建立项目目录感；
- 帮助识别模块、组件、服务之间的静态依赖关系；
- 降低“完全从代码盲读”的成本。

因此，UML 的价值是明确的，应继续保留。  
但 UML 的作用边界也非常明显：它更适合表达**结构**，不适合表达以下内容：

- 某个能力的统一行为；
- 多入口必须保持一致的规则；
- 历史决策与产品语义；
- 边界条件与例外；
- 某次修复为什么要同时影响多个模块。

结论是：

> UML 是项目地图，不是项目宪法。

它可以帮助 AI 快速知道“去哪里看”，但不能独立回答“哪些地方必须一起改、哪些差异是 bug”。

### 3.2 单一事实源文档的价值

`E:\Project\Writer\docs\ai-development-single-source-of-truth.md` 提出的方向是正确的。  
它抓住了真正关键的问题：AI 不应围绕“某个入口”理解系统，而应围绕“某个能力”理解系统。

该方案的优点在于：

- 把能力作为第一对象，而不是 UI 入口；
- 把行为定义、约束条件、影响范围、决策来源显式化；
- 把隐含在对话和代码中的工程知识抽离成可复用资产；
- 有助于让不同版本、不同 agent 共享同一套项目共识。

但该文档目前更像一份“完整治理框架设计”，如果直接整体落地，存在执行成本偏高的问题。  
对于当前项目，更适合的是：**保留核心思想，做轻量化裁剪后实施。**

---

## 4. 落地目标

本方案不追求构建庞大的治理体系，而是聚焦一个更实际的目标：

> 让冷启动 agent 在较低阅读成本下，快速恢复某项能力的核心共识，并在修改时自动意识到相关入口、相关约束和相关影响面。

为实现这一目标，建议围绕前述三层原因，分别落地。

---

## 5. 落地方案

### 5.1 针对“知识类型错配”：建立能力级文档，而非入口级文档

建议在项目中逐步建立**能力文档（Capability Doc）**，而不是继续主要围绕页面、按钮、菜单去写说明。

能力文档应聚焦回答以下问题：

1. 这个能力是什么；
2. 这个能力有哪些触发入口；
3. 这些入口哪些行为必须一致；
4. 目标目录、启用禁用、状态变化等核心规则是什么；
5. 有哪些明确边界条件；
6. 修改这个能力时通常需要检查哪些文件和模块。

为控制维护成本，不建议一开始为所有能力建档。应优先覆盖以下“高频 + 多入口 + 易分叉”能力：

- 新建文件 / 新建文件夹；
- 删除；
- 重命名；
- 拖拽；
- 工作区切换；
- 标签页相关操作。

这些能力一旦有统一文档，后续 agent 在接到类似任务时，不需要先通过阅读大量代码来猜测规则。

### 5.2 针对“知识载体错配”：建立最小化单一事实源

建议不要一次性完整落地大型 SSOT 框架，而是先落地三类最小文档：

#### A. 能力清单

记录项目当前有哪些核心能力，每项能力对应哪些入口。  
该文档适合作为冷启动入口，让 agent 先快速知道：

- 这是不是已存在能力；
- 是否只是新增入口；
- 是否会影响其他入口。

#### B. 能力详情页

每个高风险能力一页，记录：

- 行为定义；
- 统一约束；
- 关键边界；
- 影响面；
- 代码落点。

该文档是避免 AI 只看局部代码的核心载体。

#### C. 决策日志

只记录真正重要的产品 / 交互 / 架构决策。  
不要求把所有讨论都记进去，而是聚焦：

- 为什么这样统一；
- 为什么某个入口被删除；
- 为什么某些行为必须保持一致；
- 为什么某种实现方式被否决。

这样后续 agent 不需要从历史对话中反推原因。

### 5.3 针对“工作流错配”：把查阅与回写纳入固定开发流程

仅有文档还不够，必须让其进入 agent 的工作流。

建议在 `AGENTS.md` 或等价执行约束中明确：

#### 开发前

若任务涉及已有能力，必须先：

1. 查询能力清单；
2. 读取对应能力文档；
3. 判断是否影响多个入口；
4. 若涉及统一行为或约束变更，先汇报影响面，再实施。

#### 开发后

若本次修改影响了能力行为、入口集合或一致性约束，必须：

1. 更新能力文档；
2. 必要时更新决策日志；
3. 补对应能力级回归测试。

这样文档不再只是“参考资料”，而是成为改动链路中的前置和后置动作。

---

## 6. 建议的轻量实施顺序

为避免制度过重，建议分三步推进。

### 第一步：先做最小骨架

先只建立：

- 一份能力总索引；
- 三到五个高风险能力详情页；
- 一份简洁的决策日志目录。

目标不是立刻覆盖全项目，而是先覆盖最容易反复出问题的能力。

### 第二步：把 agent 工作流接上

在 `AGENTS.md` 中补充一条简单但明确的规则：

> 涉及已有能力时，先查能力文档；若影响统一行为，必须同步检查所有入口与相关约束。

这一步的作用比多写十份文档更大，因为它直接改变执行方式。

### 第三步：逐步补能力级测试

后续测试策略应从“局部函数测试”扩展到“能力一致性测试”。  
例如对“新建”能力，不只验证某个函数返回值，而应验证：

- 多入口在相同选择态下行为一致；
- 根目录与子目录场景一致；
- 启用 / 禁用规则一致；
- 折叠目录自动展开规则一致。

当文档、流程、测试三者形成闭环时，能力分叉的概率会显著下降。

---

## 7. 预期效果

若按本方案推进，预期能够带来以下改善：

1. 冷启动 agent 不再完全依赖代码逆向推理项目规则；
2. 同一能力在不同入口上的实现更容易保持一致；
3. 历史决策不再主要存活于对话上下文中；
4. 后续迭代时，新增入口更倾向于复用既有能力而不是重新发明一套逻辑；
5. 项目中的结构图（UML）与能力文档形成互补：前者负责“结构理解”，后者负责“规则理解”。

---

## 8. 结论

本次问题的根本原因，并不在于 AI 不能看懂代码，而在于：

- 项目中沉淀的是结构知识，缺少行为与决策知识；
- 关键上下文分散在不稳定载体中；
- 工作流没有强制 AI 在改动前后查询与回写这些知识。

因此，解决方案不应寄希望于单一手段。  
更可行的路径是：

1. 保留 UML 作为结构地图；
2. 建立轻量化的能力文档与决策日志作为单一事实源；
3. 通过工作流约束，让 agent 在实现前查询、在实现后回写；
4. 逐步补齐能力级一致性测试。

这套方案的目标不是消除所有 AI 误差，而是让项目具备一种更稳定的**工程记忆机制**，从而在多版本、多 agent 的持续迭代中，尽量维持行为一致性与决策可追溯性。
