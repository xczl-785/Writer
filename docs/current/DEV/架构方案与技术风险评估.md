## 一、架构建设概览

根据产品侧 (PM) 圈定的交互重构范围，对于“交互全面设计”及“结构感知架构基础”两大主线，前端团队需构建一套高内聚、易维护的交互底层设施。

## 二、开发交付阶段建议 (功能依赖与排期探索)

**建议的分阶段打桩路径（原功能排期探索）：**

```text
阶段 1 (基础设施构建)
├── INT-001 交互规范定义落库
└── INT-002 右键菜单系统底层架构搭建
         ↓
阶段 2 (核心功能落地)
├── INT-010 文件树右键菜单
├── INT-011 编辑器右键菜单
├── INT-012 表格右键菜单
├── INT-013 删除确认统一流
├── INT-014 保存状态底层指令与通信

         ↓
阶段 3 (体验与增强集成)
└── INT-003 状态反馈系统 (所有零散模块的 UI 层级接管与整合)
```

## 三、识别的重大技术风险与依赖

### 3.1 技术依赖链条确认

| 核心依赖项         | 当前可行性状态 | 潜在构建风险 |
| ------------------ | -------------- | ------------ |
| TipTap 菜单扩展 API| 待评估 (Spike) | 中           |
| TipTap 选区装饰器  | 原理上可用     | 低           |
| React Portal 挂载  | 稳健且可用     | 低           |

### 3.2 识别的业务架构风险与预案

| 已知风险问题点             | 对整体项目的影响程度 | 预案与缓解措施建议 (Mitigation Plan)    |
| -------------------------- | -------------------- | --------------------------------------- |
| **TipTap 扩展与现有代码冲突** | 致命 (高)            | **前置条件**：任何底层替换，必须先在 `spike/` 分支设立沙盒验证技术方案，切忌直接在主干上大修大补。 |
| **右键菜单与系统菜单事件冲突**| 脱轨 (中)            | **实现策略**：若 Web 端模拟右键被浏览器原生菜单接管，建议尝试使用 Tauri 原生 API 降级拦截而非纯 Web 强行覆盖。   |
| **大纲解析器的性能塌陷**      | 缓慢 (中)            | **架构硬性要求**：针对长文档处理，必须从 Day 1 起就设计好**增量解析 (Incremental parsing)** + **虚拟滚动 (Virtual scrolling)** 的双重架构设计。 |
| **跨平台的菜单交互一致性**    | 体验落差             | 统一采用 Web 组件渲染核心菜单层栈。 |
| **全局快捷键碰撞与抢占**      | 用户困惑             | **机制设立**：必须在 `src/` 中建立统一的 `ShortcutRegistry` (快捷键注册中心) 并加入冲突检测报警与拦截器。 |

## 四、功能实现的技术攻坚清单 (由 PM 前置抛出，待开发确认解答)

前端架构师及开发介入后，需针对下述业务模块的核心技术要点出具落地编码方案：

### 4.1 右键菜单系统底层 (INT-002)
- **待攻坚**：TipTap 内核右键菜单扩展能力调研。
- **待输出**：菜单渲染统一架构设计、菜单项注册机制（Registry）以及各种上下文感知识别逻辑（选区 vs 空白区）。

### 4.2 表格增强操作 (INT-012)
- **待攻坚**：拦截并计算 TipTap 表格（Table）的精确选区逻辑。

### 4.3 大纲导航面板 (INT-015)
- **待攻坚**：超长文档（>5000 行）结构的实时增量解析机制。
- **待输出**：面板虚拟滚动（Virtual Scroll）实现方案与 UI。

### 4.4 错误处理管道规范化 (INT-016)
- **待攻坚**：确定本次 V5 的标准错误机制与现有系统的老 `ErrorService` 是如何共存乃至最终融合。
- **待输出**：全局统一的报错消息结构体设计、展示层规范化 API。
