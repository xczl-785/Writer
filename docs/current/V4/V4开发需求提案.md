# V4 开发需求提案（评审稿）

- 版本：Draft v0.1
- 状态：待评审
- 更新时间：2026-02-18

## 1. 提案背景

### 1.1 V1-V3 回顾

| 版本 | 主题       | 成果                               |
| ---- | ---------- | ---------------------------------- |
| V1   | 核心能力   | 图片粘贴、自动保存、文件树交互     |
| V2   | 功能补齐   | 表格编辑、顶部菜单、搜索、查找替换 |
| V3   | 稳定性增强 | 破坏性操作一致化、表格边界修复     |

### 1.2 技术债务积累

在快速迭代过程中，代码库积累了以下技术债务：

1. **Editor.tsx 过于庞大**：1319 行代码，职责过多
2. **状态管理分散**：多个 store 同步逻辑散落各处
3. **错误处理不一致**：缺乏统一策略
4. **硬编码常量**：魔法数字散落各处

### 1.3 重构必要性

| 问题            | 影响               | 不重构的后果       |
| --------------- | ------------------ | ------------------ |
| Editor.tsx 过大 | 难以维护、难以测试 | 新功能开发效率下降 |
| 状态管理分散    | 逻辑重复、难以追踪 | Bug 定位困难       |
| 错误处理不一致  | 用户体验不一致     | 问题排查困难       |
| 硬编码常量      | 难以调整配置       | 维护成本高         |

## 2. V4 目标建议

### 2.1 Must（必须纳入）

1. **Editor 组件拆分**：将 1319 行的 Editor.tsx 拆分为多个职责单一的模块
2. **状态管理统一**：创建 actions 层封装跨 store 操作
3. **错误处理标准化**：建立统一的错误处理服务
4. **常量提取**：将硬编码值提取到配置文件

### 2.2 Should（建议纳入）

1. **测试覆盖提升**：核心模块测试覆盖率 > 80%
2. **代码注释补充**：核心逻辑添加注释
3. **类型定义优化**：消除 any 类型

### 2.3 Could（储备项）

1. **性能基准测试**：建立性能基准
2. **代码复杂度监控**：引入 ESLint 复杂度规则

### 2.4 建议不做

1. **新功能开发**：V4 纯重构，不新增功能
2. **UI/UX 变更**：保持现有交互不变
3. **依赖升级**：不升级主要依赖版本
4. **大规模架构调整**：仅做模块拆分，不改架构

## 3. 重构方案

### 3.1 Editor 组件拆分方案

#### 当前结构

```
src/ui/editor/
├── Editor.tsx          # 1319 行，包含所有逻辑
├── Editor.css
├── pasteHandler.ts
└── useImagePaste.ts
```

#### 目标结构

```
src/ui/editor/
├── Editor.tsx                    # 主容器（~200行）
├── Editor.css
├── components/
│   ├── Toolbar.tsx               # 工具栏组件
│   ├── ToolbarButton.tsx         # 工具栏按钮
│   ├── ToolbarGroup.tsx          # 工具栏分组
│   ├── FindReplacePanel.tsx      # 查找替换面板
│   └── InsertTablePopover.tsx    # 表格插入弹窗
├── hooks/
│   ├── useFindReplace.ts         # 查找替换逻辑
│   ├── useToolbarCommands.ts     # 工具栏命令
│   ├── useEditorShortcuts.ts     # 快捷键绑定
│   ├── useEditorState.ts         # 编辑器状态同步
│   └── useTableOperations.ts     # 表格操作
├── types.ts                      # 类型定义
├── constants.ts                  # 常量定义
├── pasteHandler.ts               # 保持不变
└── useImagePaste.ts              # 保持不变
```

#### 拆分原则

1. **单一职责**：每个组件/hook 只负责一件事
2. **最小改动**：保持功能不变，仅调整结构
3. **渐进式**：分阶段拆分，每阶段可独立验证

### 3.2 状态管理重构方案

#### 当前问题

```typescript
// Sidebar.tsx 中直接调用多个 store
useWorkspaceStore.getState().renameFile(...)
useEditorStore.getState().renameFile(...)
useFileTreeStore.getState().setNodes(...)
```

#### 目标方案

```typescript
// 创建统一的 actions 层
// src/state/actions/fileActions.ts
export const fileActions = {
  async renameFile(oldPath: string, newPath: string) {
    await FsService.renameNode(oldPath, newPath);
    useWorkspaceStore.getState().renameFile(oldPath, newPath);
    useEditorStore.getState().renameFile(oldPath, newPath);
    // ... 其他同步操作
  },
};

// 组件中使用
fileActions.renameFile(oldPath, newPath);
```

### 3.3 错误处理标准化方案

```typescript
// src/services/error/ErrorService.ts
export const ErrorService = {
  handle(error: unknown, context: string): void {
    const message = error instanceof Error ? error.message : String(error);
    useStatusStore.getState().setStatus('error', `${context}: ${message}`);
    console.error(`[${context}]`, error);
  },
};

// 使用示例
try {
  await FsService.deleteNode(path);
} catch (error) {
  ErrorService.handle(error, 'Failed to delete file');
}
```

### 5.1 代码门禁

1. `npm run test` 全部通过
2. `npm run build` 无错误
3. `npm run lint` 无错误
4. Editor.tsx 主文件 < 300 行

1. 所有 V3 功能正常工作
