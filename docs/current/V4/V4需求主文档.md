# V4 需求主文档

- 版本：V4
- 状态：规划中
- 最后更新时间：2026-02-18

## 1. 目标与价值

V4 是技术债务清理版本，聚焦"可维护性 + 可测试性 + 可扩展性"，不新增功能特性。

### 1.1 核心目标

1. **可维护性**：降低单文件复杂度，提升代码可读性
2. **可测试性**：拆分后模块更易独立测试
3. **可扩展性**：为后续功能迭代奠定良好架构基础

### 1.2 价值主张

| 维度            | 现状     | 目标               |
| --------------- | -------- | ------------------ |
| Editor.tsx 行数 | 1319 行  | < 300 行（主文件） |
| 单元测试覆盖    | 部分覆盖 | 核心模块 100%      |
| 状态管理        | 分散同步 | 统一 action 层     |
| 错误处理        | 不一致   | 标准化策略         |

## 2. 范围边界

### 2.1 范围内

#### FR-V4-01：Editor 组件拆分

将 `Editor.tsx` 拆分为多个职责单一的模块：

```
src/ui/editor/
├── Editor.tsx                    # 主容器（~200行）
├── components/
│   ├── Toolbar.tsx               # 工具栏组件
│   ├── ToolbarButton.tsx         # 工具栏按钮
│   ├── ToolbarGroup.tsx          # 工具栏分组
│   ├── FindReplacePanel.tsx      # 查找替换面板
│   └── InsertTablePopover.tsx    # 表格插入弹窗
├── hooks/
│   ├── useFindReplace.ts         # 查找替换逻辑
│   ├── useToolbarCommands.ts     # 工具栏命令
│   ├── useEditorShortcuts.ts     # 快捷键绑定
│   ├── useEditorState.ts         # 编辑器状态同步
│   └── useTableOperations.ts     # 表格操作
├── types.ts                      # 类型定义
└── constants.ts                  # 常量定义
```

#### FR-V4-02：状态管理统一

创建统一的 action 层，封装跨 store 操作：

```
src/state/
├── index.ts                      # 统一导出
├── actions/
│   ├── fileActions.ts            # 文件操作 action
│   ├── editorActions.ts          # 编辑器操作 action
│   └── workspaceActions.ts       # 工作区操作 action
├── slices/
│   ├── workspaceSlice.ts         # 保持不变
│   ├── filetreeSlice.ts          # 保持不变
│   ├── editorSlice.ts            # 保持不变
│   └── statusSlice.ts            # 保持不变
└── types.ts                      # 保持不变
```

#### FR-V4-03：错误处理标准化

建立统一的错误处理策略：

```typescript
// src/services/error/ErrorService.ts
export const ErrorService = {
  handle(error: unknown, context: string): void {
    const message = error instanceof Error ? error.message : String(error);
    useStatusStore.getState().setStatus('error', `${context}: ${message}`);
    // 可选：发送到日志服务
  },
};
```

#### FR-V4-04：常量提取

将硬编码值提取到配置文件：

```typescript
// src/config/editor.ts
export const EDITOR_CONFIG = {
  autosave: {
    debounceMs: 800,
  },
  table: {
    minRows: 1,
    maxRows: 20,
    minCols: 1,
    maxCols: 20,
    defaultRows: 3,
    defaultCols: 3,
  },
  findReplace: {
    maxMatches: 1000,
  },
} as const;
```

### 2.2 范围外

1. **新功能开发**：V4 不新增任何功能特性
2. **UI/UX 变更**：保持现有交互和视觉不变
3. **依赖升级**：不升级主要依赖版本
4. **性能优化**：除非重构自然带来的性能提升
5. **API 变更**：不改变对外接口

### 2.3 变更规则

1. 范围变更必须同步更新 `docs/current/V4/V4需求追溯矩阵.md`
2. 影响 Sprint 目标的变更必须同步更新 `docs/current/V4/V4启动看板.md`
3. 任何重构必须保证现有测试通过

## 3. 完成标准（门禁）

### 3.1 代码门禁

1. `npm run test` 全部通过
2. `npm run build` 无错误
3. `npm run lint` 无错误
4. Editor.tsx 主文件 < 300 行

### 3.2 测试门禁

1. 现有测试 100% 通过
2. 新增模块有对应测试
3. 核心逻辑测试覆盖率 > 80%

### 3.3 功能门禁

1. 所有 V3 功能正常工作
2. 无回归问题
3. 用户无感知变化

### 3.4 文档门禁

1. `docs/current/V4` 与 `docs/archive` 链接无断链
2. 代码注释覆盖核心逻辑

## 5. 技术/设计/工程基线

### 5.1 技术基线

- 前端：React + TypeScript + Vite（保持不变）
- 编辑器：TipTap（保持不变）
- 桌面：Tauri（保持不变）

### 5.2 设计基线

1. 保持现有视觉与交互一致性
2. 组件拆分遵循单一职责原则
3. hooks 遵循关注点分离原则

### 5.3 工程基线

1. 按 Sprint 顺序推进：设计 -> 实现 -> 测试 -> 归档
2. 每次提交必须保证测试通过
3. 重构前后功能对等验证
