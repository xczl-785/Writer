# 设计方案：全要素结构感知系统 (Structure Awareness System)

**状态**: 走查完成，待补充阻塞项 (Walked, Pending Fixes)
**作者**: AI Product Manager
**日期**: 2026-02-17
**版本**: v2.0

> **设计哲学：分层感知 (Layered Awareness)**
>
> 1. **宏观结构 (Macro)**：通过 **左侧装订线 (Gutter)** 提示块级元素类型。
> 2. **微观样式 (Micro)**：通过 **原地展开 (Inline Expansion)** 处理行内样式（Typora 经典模式）。
> 3. **复杂容器 (Container)**：通过 **增强型浮动工具 (Floating Tools)** 处理表格/公式/图表。

## 1. 背景与问题定义 (Context)

### 1.1 核心痛点

在 "所见即所得" (WYSIWYG) 编辑器中，Markdown 源码被隐藏，导致用户在**浏览**或**编辑**长文档时，容易产生 **"语境迷失" (Context Loss)**：

- 难以区分外观相似的块级元素（如：H3 vs 加粗文本，H1 vs H2）。
- 在深层嵌套列表或引用块中，无法直观判断当前缩进层级。
- 缺乏对文档结构的全局感知和快速调整能力。

### 1.2 目标用户画像

**知识管理者 (Knowledge Manager)**

- **特征**：不仅关注写作内容，更关注内容的**结构化**。经常需要调整段落顺序、升级/降级标题、转换块类型。
- **需求**：
  1.  **清晰度**：一眼识别当前块的类型（是 H2 还是 H3？）。
  2.  **可控性**：能快速选中整个块进行移动或转换。
  3.  **沉浸感**：不希望被过多的 UI 元素打扰写作心流。

## 3. 全要素交互规范 (Interaction Specs)

### 3.1 第一层：块级元素 (Block Elements)

**策略**：左侧装订线提示 (Gutter Indicator) —— 方案 A
**触发**：鼠标悬停 (Hover) 或 光标激活 (Active)

| 元素类型              | 视觉表现 (Visual)                              | 交互行为 (Behavior)             |
| :-------------------- | :--------------------------------------------- | :------------------------------ |
| **标题 (H1-H6)**      | 装订线显示淡灰色标签 `H1` `H2` ... `H6`        | 悬停/光标激活时淡入，离开时淡出 |
| **引用 (Blockquote)** | 引用竖线加粗/变色，装订线显示 `"`              | 支持嵌套层级视觉区分            |
| **列表 (List)**       | 装订线显示连接线 (Guide Line) 指向父级         | 悬停圆点/序号时高亮整块         |
| **代码块 (Code)**     | 右上角显示语言标签（如 `JS`），装订线显示 `<>` | 点击右上角可复制代码            |
| **水平线 (HR)**       | 悬停时线条变粗，光标变为指针                   | 点击可选中删除                  |
| **任务列表 (Task)**   | 保持原样（复选框本身即为最佳提示）             | —                               |

### 3.2 第二层：行内样式 (Inline Styles)

**策略**：Typora 经典模式 (Source Expansion) ✅ 已确认
**触发**：光标进入 (Focus In) / 光标移出 (Focus Out)

| 元素类型          | 非编辑态 (Blur)  | 编辑态 (Focus)    | 备注                                |
| :---------------- | :--------------- | :---------------- | :---------------------------------- |
| **加粗 (Bold)**   | **加粗文本**     | `**加粗文本**`    | 保持纯文本编辑的流畅感              |
| **斜体 (Italic)** | _斜体文本_       | `*斜体文本*`      | 解决斜体在中文下难以辨识的问题      |
| **行内代码**      | `Code`           | `` `Code` ``      | 保持等宽字体对齐                    |
| **链接 (Link)**   | 蓝色链接文本     | `[链接文本](URL)` | 悬停时在下方显示微型 URL Tooltip    |
| **图片 (Image)**  | 显示渲染后的图片 | `![Alt](src)`     | 悬停时图片上方显示"缩放/对齐"工具栏 |

### 3.3 第三层：复杂容器 (Advanced Containers)

**策略**：增强型浮动工具 (Enhanced Floating Tools)

| 元素类型         | 交互规范                                                                |
| :--------------- | :---------------------------------------------------------------------- |
| **表格 (Table)** | 悬停时：上方显示列操作柄 (Column Handle)，左侧显示行操作柄 (Row Handle) |
| **数学公式**     | 点击渲染后的公式 → 弹出 LaTeX 编辑浮层                                  |
| **Mermaid 图表** | 悬停 → 右上角显示 `Edit` 按钮 → 点击进入代码模式                        |

## 4. 设计方案选项 (Design Options)

针对 **Writer (Tiptap + React)** 技术栈，提出以下三种方案（**已选定方案 A 作为 V2.0 起步**）：

### 方案 A：极简装订线提示 (The "Ghost" Indicator) - 推荐起步

**核心理念**：仅在需要时出现，通过视觉暗示而非强制交互。

- **交互设计**：
  - **悬停/激活**：当光标位于 H1-H6、引用、列表项时，编辑器左侧装订线 (Gutter) 区域淡入显示对应的灰色标签（如 `H1`, `”`, `•`）。
  - **当前行高亮**：光标所在行，装订线处显示一条细微的色条，色条颜色或缩进深度代表层级。
- **优点**：开发成本低，视觉干扰最小，保留了 Typora 的沉浸感。
- **缺点**：功能单一（仅提示），无法进行复杂操作。

### 方案 B：交互式块手柄 (The "Block Handle") - 推荐进阶

**核心理念**：结构可视化与操作的结合 (Notion Style)。

- **交互设计**：
  - **悬停显形**：鼠标悬停在段落上时，左侧 `padding` 区域出现 `⋮⋮` 图标。
  - **点击菜单**：点击图标弹出菜单：
    - **类型转换**：Turn into H1/H2/List...
    - **操作**：Delete, Duplicate, Copy Link...
  - **拖拽移动**：按住图标可拖动整块内容排序。
- **优点**：赋予用户极强的结构掌控力，完美契合"知识管理"需求。
- **缺点**：实现复杂度高 (需自定义 Tiptap NodeView)，可能与纯文本写作流冲突。

### 方案 C：上下文面包屑 (The "Context" Breadcrumb)

**核心理念**：全局定位导航。

- **交互设计**：
  - **顶部常驻**：编辑器顶部（Sticky Header）显示当前光标路径：`文档 > H1 标题 > H2 子标题 > 当前段落`。
  - **点击跳转**：点击面包屑节点可跳转或折叠。
- **优点**：解决长文档迷路问题，不占用正文横向空间。
- **缺点**：视线需上下跳跃，对视线流有打断。

## 5. 场景覆盖报告 (Scenario Coverage Report)

### 5.1 走查方法

- 来源一：逐条展开文档规格为可执行的用户操作序列。
- 来源二：通用场景清单（空状态、极端状态、状态切换、冲突状态、纯键盘、性能、首次使用）。

### 5.2 覆盖统计

- 总场景数：34
- 已覆盖：4 (12%)
- 未覆盖：26 (76%)
- 待确认：4 (12%)

### 5.3 阻塞性问题 (Blocking)

1. 光标激活 + 装订线标签同时出现导致信息冗余（需定义优先级或去重规则）。
2. 混合块类型（如引用块内标题）装订线显示优先级未定义。
3. 纯键盘用户无法触发 Hover 提示（需定义键盘等价触发方式）。
4. 行内展开导致行宽跳动（渲染态与源码态切换时需要稳定策略）。

### 5.4 结论

当前设计可作为方向基线，但在上述阻塞项未补齐前，不建议直接进入开发执行。

## 6. 推荐实施路径 (Recommendation)

鉴于 **Writer** 目前处于 V2 阶段，且核心是 **Tiptap**，建议分两步走：

### 第一阶段 (V2.0 - 快速验证)

**实施方案 A (极简装订线)**

- **实现方式**：利用 Tiptap 的 `Decoration` 或简单的 CSS 伪元素。
- **目标**：以最小成本解决"我在哪"的痛点，不引入复杂的拖拽逻辑。
- **具体表现**：
  - H1-H6 左侧显示淡淡的 `H1`...`H6` 字符。
  - 引用块左侧显示竖线增强。

### 第二阶段 (V2.x - 结构增强)

**升级至方案 B (交互手柄)**

- **实现方式**：引入 Tiptap 的 `DragHandle` 扩展或自定义 React NodeView。
- **目标**：满足"知识整理"需求，允许拖拽重排和快速类型转换。

## 7. 技术可行性 (Tiptap)

- **Tiptap Extension-Gapcursor**: 已有基础。
- **Draggable Block**: Tiptap 官方有示例 (`drag-handle-plugin`)，React 社区也有成熟实现。
- **NodeView**: React 组件可以直接作为 NodeView 渲染，实现方案 B 的菜单非常方便。
