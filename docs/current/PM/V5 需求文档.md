## 第一部分：顶层设计

### 1.1 软件风格定义

#### 产品人格

Writer 的产品人格可概括为：**沉静、专注、可信赖的写作伙伴**。

| 维度     | 定位     | 说明                           |
| -------- | -------- | ------------------------------ |
| **气质** | 沉静内敛 | 不炫技、不打扰，让内容成为焦点 |
| **响应** | 即时流畅 | 操作反馈迅速但不突兀           |
| **智能** | 隐性辅助 | 功能可发现但不主动打断         |
| **可靠** | 稳定一致 | 行为可预测，无意外状态跳转     |

#### 设计哲学

> **"无打断写作主流程优先"**

所有交互设计必须通过以下测试：

1. 这个功能是否会打断用户写作心流？
2. 这个交互是否增加了认知负担？
3. 这个反馈是否必要且时机恰当？

#### 竞品定位矩阵

| 维度         | Typora             | Obsidian         | Writer 定位         |
| ------------ | ------------------ | ---------------- | ------------------- |
| **视觉风格** | 极简所见即所得     | 技术感双栏       | 同 Typora，更温润   |
| **文件管理** | 单文件打开         | 库式管理         | 同 Obsidian，更轻量 |
| **交互密度** | 低（隐藏式工具栏） | 高（常驻侧边栏） | 中（可折叠侧边栏）  |
| **可定制性** | 低                 | 高（插件生态）   | 中（精选功能）      |
| **学习曲线** | 平缓               | 陡峭             | 平缓 + 渐进式发现   |

### 1.2 用户故事体系

#### 角色定义

| 角色           | 特征                         | 核心诉求                     |
| -------------- | ---------------------------- | ---------------------------- |
| **深度写作者** | 长文创作、学术研究、小说写作 | 专注、无打扰、结构清晰       |
| **知识管理者** | 笔记整理、知识卡片、第二大脑 | 快速检索、双向链接、标签体系 |
| **轻量记录者** | 日常笔记、临时想法、待办清单 | 快速启动、多端访问（Windows/Mac/Linux，本地文件，不负责数据同步） |

### 1.3 交互原则

#### 第一原则：无打断 (Non-Disruption)

**定义**: 任何交互不应打断用户的写作心流。

**实施细则**:

- 非紧急反馈采用被动展示（状态栏、角标）而非主动弹窗
- 自动保存无声执行，仅在有状态变化时更新指示器
- 错误处理优先尝试自动恢复，无法恢复时再提示用户

**反模式**:

- ❌ 自动弹出更新提示
- ❌ 模态对话框阻断编辑
- ❌ 保存成功 Toast 弹窗

---

#### 第二原则：一致性 (Consistency)

**定义**: 相同操作在所有场景下表现一致。

**实施细则**:

- 所有破坏性操作统一使用二次确认
- 所有文件操作统一在右键菜单提供入口
- 所有快捷键在 UI 上有可发现的提示路径
- 所有状态变化有视觉反馈

**反模式**:

- ❌ 某些删除需要确认，某些不需要
- ❌ 某些地方右键有菜单，某些地方没有
- ❌ 快捷键隐藏且无查询路径

---

#### 第三原则：可发现 (Discoverability)

**定义**: 功能不应完全隐藏，用户应有路径发现和学习。

**实施细则**:

- 右键菜单提供常用功能入口
- 悬停提示显示快捷键
- 提供快捷键速查面板（Mod+/ 或 Mod+Shift+/）
- 新功能首次可用时有温和引导（非模态）

**反模式**:

- ❌ 功能仅可通过快捷键触发且无提示
- ❌ 首次使用无任何引导
- ❌ 设置隐藏在深层菜单中

---

#### 第四原则：渐进式 (Progressive)

**定义**: 交互密度应允许用户从简单到复杂逐步学习。

**实施细则**:

- 基础功能（新建/保存/删除）直观可见
- 进阶功能（格式/插入）通过右键菜单发现
- 高级功能（自定义/扩展）在设置中提供
- 不强制用户学习全部功能即可开始使用

**反模式**:

- ❌ 首次启动显示全部复杂功能
- ❌ 不使用教程无法完成基础操作
- ❌ 设置项混杂基础和高级选项

| 功能         | 排除理由                     |
| ------------ | ---------------------------- |
| 实时协作编辑 | 偏离个人写作工具定位         |

| 插件系统     | 增加复杂度，违背精选功能原则 |
| AI 辅助写作  | 偏离工具定位，可能打扰心流   |

## 第三部分：验收框架

### 3.1 用户故事验收 (权重 50%)

**方法**: 每个用户故事按完成度评分

| 完成度       | 定义                                 | 得分 |
| ------------ | ------------------------------------ | ---- |
| **完全实现** | Given-When-Then 全部通过，无已知缺陷 | 100% |
| **基本实现** | 主流程通过，边缘情况有缺陷           | 70%  |
| **部分实现** | 功能可用但有明显体验问题             | 40%  |
| **未实现**   | 功能未开发或不可用                   | 0%   |

**V5 通过标准**: 所有 Must 优先级用户故事 ≥ 90 分，Should 优先级 ≥ 70 分

---

### 3.2 功能清单验收 (权重 30%)

**方法**: 按功能项技术验收点逐项检查

| 验收状态    | 定义                           |
| ----------- | ------------------------------ |
| ✅ 通过     | 所有技术验收点满足             |
| ⚠️ 部分通过 | 主功能可用，非关键验收点未满足 |
| ❌ 未通过   | 主功能不可用或有严重缺陷       |

**V5 通过标准**: 所有 Must 功能 ✅ 通过，Should 功能 ≥ 80% ✅ 通过

---

### 3.3 质量属性验收 (权重 20%)

**方法**: 按质量指标度量

| 指标                 | 目标值           | 测量方法                      |
| -------------------- | ---------------- | ----------------------------- |
| **交互一致性**       | ≥ 90 分          | 操作行为一致性审查清单        |
| **快捷键冲突率**     | 0                | 快捷键映射表审查              |
| **破坏性操作覆盖率** | 100%             | 删除/覆盖类操作确认对话框审查 |
| **错误信息可读性**   | ≥ 80 分          | 错误信息包含原因 + 建议的比例 |
| **性能无感知**       | 操作响应 < 100ms | 关键交互性能测试              |

**V5 通过标准**: 所有指标达到目标值

## 第四部分：风险与依赖

### 4.1 技术风险

| 风险                       | 概率 | 影响 | 缓解措施                          |
| -------------------------- | ---- | ---- | --------------------------------- |
| TipTap 扩展与现有代码冲突  | 中   | 高   | 先在 spike/ 分支验证技术方案      |
| 右键菜单与系统菜单冲突     | 低   | 中   | 使用 Tauri 原生 API 而非 Web 实现 |
| 大纲解析性能问题（长文档） | 中   | 中   | 实现增量解析 + 虚拟滚动           |

### 4.2 依赖关系

```
交互规范文档 → 右键菜单框架 → 文件树/编辑器右键菜单
                     ↓
              快捷键提示系统
                     ↓
              大纲导航面板 ← 文档结构解析
                     ↓
              状态反馈优化
```

### 4.3 决策点

| 决策点         | 触发条件          | 决策内容                              |
| -------------- | ----------------- | ------------------------------------- |
| 结构感知去留   | 阶段 2 完成后     | 基于交互设计进度决定纳入 V5 或推迟 V6 |
| 面包屑导航去留 | 大纲导航完成后    | 评估导航需求满足度决定是否需要        |
| 专注模式优先级 | V5 核心功能完成后 | 评估剩余时间决定是否纳入              |
