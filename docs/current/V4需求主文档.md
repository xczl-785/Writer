### 2.3 遗留项处置规则（补遗漏）

V4 制定时存在未完成功能/遗留问题。处理原则：

1. **阻塞重构主线**的遗留项可纳入 V4（需在 `V4执行与证据包.md` 记录）。
2. **不阻塞主线**的遗留项不混入 V4 FR，转入下一阶段需求池并留档。
3. 任何临时插入项必须满足：不改变 V4“纯重构”定位。

参考输入：`docs/current/V3遗留与V4待办.md`、`docs/current/全流程需求跟踪台账.md`。

## 3. 工程门禁（Go/No-Go）

### 3.1 代码门禁

1. `npm run test` 全部通过。
2. `npm run build` 通过。
3. `npm run lint` 通过。
4. `wc -l src/ui/editor/Editor.tsx` < 300。

### 3.2 测试门禁

1. 回归测试全部通过。
2. 新增模块有对应测试。
3. 核心模块覆盖率 > 80%。

### 3.3 功能门禁

1. V3 核心能力无回归。
2. 用户侧交互无感知变化（功能不变）。

### 3.4 文档门禁

1. `docs/current` 与 `docs/archive` 链接无断链。
2. 追溯矩阵存在 FR -> Task -> QA -> Evidence 闭环。

## 4. 文档结构（工程化最小集合）

1. `docs/current/V4需求主文档.md`：范围、门禁、规则（本文件）。
2. `docs/current/V4重构实施计划.md`：实施顺序、任务包、风险处置。
3. `docs/current/V4执行与证据包.md`：执行检查清单与证据留存。
4. `docs/current/V4需求追溯矩阵.md`：FR 与任务/QA/证据映射。
5. `docs/current/V4启动看板.md`：阶段状态与入口索引。
