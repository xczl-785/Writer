# 新建文件与新建文件夹统一行为梳理与整改基线

## 1. 文档目的

本文用于统一梳理 Writer 当前“新建文件 / 新建文件夹”相关入口、逻辑现状、已知问题与后续整改优先级，作为后续逐项修正的基线文档。

本文关注的核心问题包括：

1. 各入口是否复用同一套创建逻辑。
2. UI 层与逻辑层是否存在耦合。
3. 各入口的禁用 / 启用状态是否一致。
4. 各入口在目标目录判定上是否一致。
5. 各入口在折叠目录场景下的展开行为是否一致。
6. 已知 Bug 是否能够纳入统一整改范围。

## 2. 当前纳入统一梳理的入口范围

当前与“新建文件 / 新建文件夹”直接相关的入口一共四类：

1. 侧边栏顶部按钮
2. 菜单栏“文件”菜单下的新建文件 / 新建文件夹
3. 右键文件夹
4. 右键根文件夹

说明：
- 右键空白区域菜单已删除，不再纳入后续统一范围。
- 文件右键中的“新建文件 / 新建文件夹”已按既定规则移除，不再作为目标形态保留。

## 3. 当前现状总览

### 3.1 顶部按钮与文件菜单并未真正走独立逻辑

当前实现中：

- 侧边栏顶部按钮在 `src/ui/sidebar/Sidebar.tsx` 内直接调用 `dispatchExplorerCommand(...)`。
- 菜单栏“文件 > 新建文件 / 新建文件夹”先在 `src/app/commands/fileCommands.ts` 中映射为 `new-file / new-folder`，再通过 `writer:sidebar-command` 事件把请求转发给 `Sidebar`，最终仍由 `Sidebar` 内部继续派发 `dispatchExplorerCommand(...)`。

这意味着：

- 从结果上看，这两个入口当前确实都收口到 `Sidebar` 组件内部。
- 从分层上看，当前的“创建编排逻辑”并未从 Sidebar UI 中抽离出来，存在明显的 UI 与逻辑耦合。

### 3.2 右键入口与顶部 / 菜单入口并未完全统一

当前右键入口已经过一轮收敛，形成以下规则：

- 右键所有文件夹：有“新建文件 / 新建文件夹”
- 右键所有根文件夹：有“新建文件 / 新建文件夹”
- 右键文件：没有“新建文件 / 新建文件夹”

但在实现层面，右键入口与顶部 / 菜单入口仍不是完全同一条逻辑链：

- 顶部 / 菜单：通过 `startCreate(...)` 主链处理
- 右键文件夹 / 根文件夹：更多是通过上下文直接设置 ghostNode，并在部分路径中补充“先展开再新建”的逻辑

这意味着产品规则已经在收敛，但实现结构仍未彻底收口。

## 4. 当前已识别的问题清单

## 4.1 分层问题：创建逻辑收口在 Sidebar 组件内

### 现状

当前创建相关的关键动作主要分布在 `src/ui/sidebar/Sidebar.tsx`：

- 创建命令接收
- 创建目标判定
- ghostNode 设置
- 折叠目录展开
- 创建后树刷新与选中更新

### 问题

这会带来以下风险：

1. 菜单入口依赖 Sidebar 是否已挂载、是否可见。
2. 业务规则无法被多个入口自然复用，只能通过 UI 事件转发复用。
3. 新增入口时，容易继续走“把请求转发给 Sidebar”的路径，进一步放大耦合。
4. 问题排查时，UI 行为问题与业务判定问题混在一起，不利于维护。

### 结论

需要将“创建动作的统一判定与执行编排”从 Sidebar UI 中抽离，形成更清晰的逻辑层。

## 4.2 状态问题：顶部按钮与菜单入口的启用 / 禁用语义并未完全统一

### 现状

侧边栏顶部按钮当前具有显式禁用态：

- `disabled={!canCreateFromWorkspace(currentPath)}`

而菜单入口当前更接近于：

- 命令可以触发
- 执行时再根据 `hasWorkspace` 做拦截
- 无法创建时通过状态提示报错

### 问题

这会形成以下不一致：

1. 顶部按钮表现为“不可点”。
2. 菜单入口可能表现为“可点但执行时报错”或“是否禁用取决于菜单层实现”。
3. 用户对两个入口会形成不同预期，不符合“同义入口行为一致”的原则。

### 交互习惯评估

对于“新建文件 / 新建文件夹”这类依赖工作区上下文的动作，更符合习惯的行为是：

- 当无可用工作区时，入口直接禁用。
- 若某些平台层限制无法实时禁用，逻辑层仍需兜底阻止执行。

### 结论

顶部按钮与菜单入口应统一为同一套“可创建判定”，并尽可能统一为同样的禁用态语义。

## 4.3 规则问题：目标目录判定逻辑仍需彻底统一

### 当前已收敛的方向

已基本形成的产品规则如下：

1. 选中根文件夹时，在该根文件夹下创建。
2. 选中文件夹时，在该文件夹下创建。
3. 选中文件时，在该文件的父目录下创建。
4. 无选中项时，在默认根目录下创建。
5. 无工作区时，不允许创建。

### 当前风险

虽然规则方向已明确，但实现层面仍需确认以下问题：

1. 顶部按钮 / 文件菜单 / 右键是否都经过同一个“目标目录解析函数”。
2. 默认根目录的选取规则是否稳定，尤其在多根工作区下是否仍会错误偏向第一个根目录。
3. 一级文件场景下，是否存在根路径与父目录解析结果不一致的问题。

### 结论

后续整改中，目标目录判定规则应抽象为单一、可测试、与 UI 无关的纯逻辑函数。

## 4.4 交互问题：折叠目录自动展开目前尚未覆盖所有新建入口

### 已有结论

现已明确的统一原则为：

> 新建操作一旦确定目标目录，若该目录当前处于折叠状态，则应先展开，再进入新建预览。

### 当前实现状态

右键文件夹 / 右键根文件夹已开始覆盖该规则。

但顶部按钮与文件菜单当前是否完整遵循这条规则，仍需作为统一整改的一部分继续处理和验证。

### 结论

“先展开，再新建预览”必须升级为所有新建入口共享的通用规则，而不是仅对右键入口生效的局部修补。

## 4.5 已知 Bug：选中根目录下一级文件时，顶部和菜单新建无响应

### 现象

当选中根目录直接子级的文件时：

- 侧边栏顶部新建文件 / 新建文件夹没有作用
- 菜单中的新建文件 / 新建文件夹也没有作用

### 初步判断

该问题与以下几类逻辑高度相关：

1. 目标目录解析
2. 根目录判定
3. `selectedPath` / `selectedNode` / `activeFile` 的优先级关系
4. 根层级 ghost 渲染位置与触发条件

### 当前处理原则

该问题不应单独零散修补，而应放在“统一逻辑抽离、统一判定、统一折叠展开”之后处理，以避免重复返工。

## 5. 建议采用的统一规则模型

建议后续把所有“新建文件 / 新建文件夹”入口统一收敛为同一模型。

## 5.1 统一输入

所有入口最终只向逻辑层提供以下输入：

1. 创建类型：`file` / `directory`
2. 当前工作区上下文
3. 当前选中项
4. 当前活动文件
5. 当前展开状态
6. 触发来源（顶部 / 菜单 / 右键 / 快捷键）

说明：
- 触发来源只用于 UI 协调、埋点或日志，不应影响目标目录判定结果。

## 5.2 统一输出

逻辑层应输出统一的创建决策结果，包括：

1. 是否允许创建
2. 目标根目录
3. 目标父目录
4. 是否需要先展开目标目录
5. 创建预览应出现的位置
6. 若不允许创建，应给出统一的禁用 / 拦截原因

## 5.3 统一判定规则

建议固化为以下优先级：

1. 选中的是根文件夹 → 在该根文件夹下创建
2. 选中的是普通文件夹 → 在该文件夹下创建
3. 选中的是文件 → 在其父目录下创建
4. 无选中项 → 在默认根目录下创建
5. 无工作区 → 不可创建

## 5.4 统一展开规则

建议固化为以下规则：

1. 一旦确定了目标目录，即检查其展开状态。
2. 若目标目录当前折叠，则先展开。
3. 展开完成后，再进入 ghost / 输入态。
4. 根级创建也应遵循同一规则。

## 5.5 统一禁用规则

建议固化为以下规则：

1. 无工作区时，所有“新建文件 / 新建文件夹”入口应统一禁用。
2. 若菜单层无法实时禁用，逻辑层仍必须阻止执行。
3. 顶部按钮、菜单入口、后续快捷键入口，都应消费同一套 `canCreate` 结果。

## 6. 建议优先级

### P0：先做统一基础能力

1. 抽离“新建动作逻辑”，实现 UI 与逻辑分离。
2. 统一所有入口的可创建判定与禁用 / 启用状态。
3. 统一所有入口的“目标目录折叠时先展开”规则。

### P1：修复已知功能缺陷

4. 修复“选中根目录下一级文件时，顶部和菜单新建无响应”的问题。

### P2：做结构收尾

5. 清理顶部按钮、菜单入口、右键入口之间的重复适配层。
6. 统一测试模型，补齐场景矩阵。

## 7. 建议测试场景矩阵

后续整改时，至少应覆盖以下场景：

1. 无工作区时：顶部按钮禁用，菜单入口禁用或被统一拦截。
2. 选中根文件夹时：在根目录下创建。
3. 选中普通文件夹时：在该文件夹下创建。
4. 选中文件时：在父目录下创建。
5. 无选中项时：在默认根目录下创建。
6. 目标目录折叠时：先展开，再显示 ghost。
7. 目标目录已展开时：直接进入 ghost。
8. 多根工作区下：目标根目录不能错误漂移到其他根。
9. 根目录下一级文件被选中时：顶部与菜单入口均能正常创建到正确父目录。
10. 顶部按钮、菜单入口、右键入口三者在同场景下行为一致。

## 8. 后续执行建议

建议后续不要按“入口”分别修，而应按“统一能力”分批推进：

1. 先抽统一创建决策逻辑。
2. 再让顶部按钮、菜单入口、右键入口共同依赖该逻辑。
3. 再补齐折叠展开与 ghost 呈现。
4. 最后处理一级文件场景下的已知 Bug。

这样可以避免：

- 修好一个入口，另一个入口再次漂移。
- 逻辑层问题被误判为 UI 问题。
- 同一规则在多个入口中各自复制一份。

## 9. 结论

当前“新建文件 / 新建文件夹”功能已经完成了一轮入口收敛，但仍存在以下核心问题：

1. 创建逻辑实际收口在 Sidebar UI 内，分层不清。
2. 顶部按钮与菜单入口的禁用 / 启用语义未彻底统一。
3. 折叠目录自动展开尚未对所有新建入口形成统一规则。
4. 选中根目录下一级文件时的新建无响应，仍是待修复的已知缺陷。

后续整改应以“统一逻辑抽离、统一判定规则、统一交互行为”为主线推进，而不应继续以单个入口为单位零散修补。
