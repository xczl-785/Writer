# V3 开发需求提案（评审稿）

- 版本：Draft v0.1
- 状态：评审通过（作为 V3 启动执行依据）
- 更新时间：2026-02-18

## 1. 提案背景

V2 已完成核心能力闭环（表格、顶部菜单、搜索、查找替换、回归与归档），当前进入 V3 启动期。
V3 的目标建议从“功能补齐”转为“稳定性与效率并重”：

1. 优先提升高频编辑链路稳定性与可恢复性。
2. 在不破坏现有心智的前提下做体验增强。
3. 继续维持可追溯交付（FR -> Task -> QA -> Evidence）。

## 2. V2 复盘输入（用于 V3 取舍）

1. V2 中高频迭代集中在编辑器与文件树，说明核心交互仍是主战场。
2. 结构感知系统在 V2 明确冻结，说明 V3 若重启需先做最小可行切片。
3. 手工验收在 Web 模式受限（原生能力触发不足），V3 需要更明确的平台测试策略。

## 3. V3 目标建议

### 3.1 Must（建议纳入 V3 主线）

1. 编辑器可恢复性增强：关键破坏性动作统一支持“可撤销、可提示、可回滚”。
2. 表格编辑闭环增强：删除、重排、边界处理（空行/空表/连续表）一致化。
3. 文件树基础操作一致性：根目录、选择态、创建目标路径、过滤规则持续稳态。
4. 回归效率提升：补齐高风险链路用例模板，降低手工回归漏测概率。

### 3.2 Should（建议进入 V3 候选）

1. 大文档查找替换性能与反馈优化。
2. 编辑器状态提示统一（成功/失败/可恢复建议）。
3. Sprint 执行证据模板标准化（减少文档重复维护）。

### 3.3 Could（建议作为储备）

1. 结构感知能力最小切片 PoC（仅提示，不做复杂交互）。
2. 平台差异化行为说明文档（Web/Tauri 的能力差异矩阵）。

## 4. 建议不做

1. 大规模架构重构。
2. 未评审依赖引入。
3. 与现有交互模型冲突的“重交互系统”一次性上线。

## 5. V3 分 Sprint 提案

## Sprint0（启动）

1. 需求收敛：冻结 Must/Should/Could 清单。
2. 追溯矩阵初始化：至少建立 FR-V3-01~FR-V3-04 骨架。
3. 测试基线：整理高风险回归清单并确认平台执行策略。

## Sprint1（核心稳定）

1. 编辑器破坏性动作一致化（含提示与撤销链路）。
2. 表格关键删除/边界场景补齐。

## Sprint2（体验增强）

1. 查找替换与结果反馈增强。
2. 文件树高频操作流畅度优化。

## Sprint3（回归与治理）

1. 回归用例执行与问题收敛。
2. 性能与异常场景治理。

## Sprint4（收尾）

1. Go/No-Go 审计。
2. 证据归档与追溯矩阵闭环。

## 6. 验收门禁建议

1. 自动化门禁：`npm run test`、`npm run build` 持续通过。
2. 手工门禁：高风险用例清单全部执行完成。
3. 缺陷门禁：P0/P1 清零，P2 有留档与修复计划。
4. 文档门禁：`docs/current` 与 `docs/archive` 链接无断链。

## 7. 风险与缓解

1. 平台能力差异导致验收偏差。
   - 缓解：明确 Web/Tauri 双口径测试项。
2. 文档分散导致维护负担。
   - 缓解：统一模板，减少重复文档。
3. 小迭代频繁导致回归成本上升。
   - 缓解：优先建设“高风险最小回归集”。

## 8. 评审输出期望

请在评审时确认以下决策：

1. V3 Must 清单是否按本提案执行。
2. Sprint1 的首批实现项优先级排序。
3. 是否将结构感知 PoC 纳入 V3（或继续冻结）。
