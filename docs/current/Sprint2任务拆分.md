# Sprint 2 Tasks（Step E）

## 1. Sprint 目标

**完成单工作区文件流闭环**。
在 Sprint 1 “可编辑”的基础上，补全“可管理”能力，使用户能在应用内完成文件/文件夹的增删改名，无需切换到操作系统资源管理器。

## 2. 范围

### In Scope（本期承诺 - 依据 V1全阶段路线图）

1.  **文件管理原子能力**：`FsService` 支持新建、重命名、删除（文件与文件夹）。
2.  **侧边栏交互**：
    - 支持上下文菜单（右键）或操作按钮。
    - 支持新建文件/文件夹交互。
    - 支持重命名交互（行内编辑或弹窗）。
    - 支持删除交互（含二次确认）。
3.  **编辑器联动**：
    - 删除当前打开文件时，编辑器应正确响应（关闭或显示为空）。
    - 重命名当前打开文件时，编辑器标题/路径应自动更新。
4.  **图标与样式优化**：引入图标库，提升侧边栏可读性。

### Out of Scope（本期不承诺）

1.  **拖拽移动**（文件树内拖拽）。
2.  **搜索**（全文搜索或文件名搜索）。
3.  **多标签页**（Tabs 仍然不在 V1 承诺范围内，维持单编辑器模式）。
4.  **图片资源管理**（明确在 Sprint 3）。

## 4. 建议执行顺序

1.  **A 组 (基建)**: S2-01 (FS扩展), S2-02 (图标/UI) —— _可以并行_。
2.  **B 组 (核心交互)**: S2-03 (新建), S2-04 (重命名), S2-05 (删除) —— _依赖 S2-01_。
3.  **C 组 (联动优化)**: S2-06 (脏状态联动) —— _依赖 B 组_。
4.  **D 组 (收口)**: S2-07 (回归)。

## 5. 风险与应对

1.  **风险**：文件重命名导致 `activeFile` 路径失效。
    - **应对**：在 `rename` 成功后，必须原子更新 `workspaceSlice` 中的 `activeFile` 和 `openFiles` 列表。

2.  **风险**：删除非空文件夹。
    - **应对**：Rust 端 `fs::remove_dir_all` 需谨慎使用，前端必须有二次确认（Confirm Dialog）。

3.  **风险**：Windows/Mac 路径分隔符差异。
    - **应对**：继续坚持在 Service 层统一处理路径格式，或依赖 Rust `std::path` 的跨平台能力。
